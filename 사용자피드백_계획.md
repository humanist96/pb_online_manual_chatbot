# 사용자 피드백 계획 — 온라인 데모 등록·조회 양식

2026-07-08. 대상: **온라인 데모(`deploy/online/`) 전용** — 사내 webapp 이식은 범위 밖(온라인 테스터 피드백 수집이 목적).

## 1. 문제 정의 / 목표

온라인 데모를 사람들에게 열어 직접 테스트하게 하는 단계인데, 테스트 중 발견한 것을 **그 자리에서 남길 통로가 없다**. 버그·품질 지적이 구두/메신저로 흩어지면 유실되고, 재현 맥락(어떤 질문에 어떤 답이 나왔는지)이 함께 남지 않아 후속 조치가 어렵다.

목표: 데모 웹페이지 안에서 피드백을 **등록**하고 **조회**할 수 있는 양식을 붙인다.

| 피드백 유형 | 예시 | 후속 조치 대상 |
|---|---|---|
| `bug` 버그/오류 | 화면 깨짐, 검색 실패, 무한 로딩 | 데모 코드 수정 |
| `quality` 답변 품질 | 부정확한 답, 근거와 다른 답, 게이트 오판 | 프롬프트·τ·검색 튜닝, 평가셋 시드 |
| `outdated` 매뉴얼 최신화 | 매뉴얼 내용이 현행 업무와 다름 | 원문 매뉴얼 갱신 요청 목록 |
| `missing` 매뉴얼 보강 | 물었는데 매뉴얼에 없는 내용 | 보강 요청 목록, 질문뱅크 확충 |
| `idea` 제안/기타 | UI 개선, 기능 요청, 칭찬 | 로드맵 반영 |

핵심 설계 가치: **품질 피드백은 재현 컨텍스트(질문·답변·근거·게이트 점수)가 자동 첨부**되어야 한다. "답이 이상해요"만 남으면 조치 불가 — 답변 말풍선에서 바로 피드백을 열면 그 턴의 전체 맥락이 붙는다. 이것이 [전부문확장_계획.md]의 교차 오염 방지 피드백 루프의 실제 입력 채널이 된다.

## 2. 설계 원칙

- **인프라 추가 없음** — 저장소는 이미 프로비저닝된 Upstash Redis(현재 일일 답변 상한·칩 계측 카운터용, `UPSTASH_REDIS_REST_URL` 설정 완료). 별도 DB·외부 폼 서비스(Google Forms 등) 불사용.
- **의존성 추가 없음** — `_common.py`의 표준 라이브러리 `_post()` REST 패턴 그대로. Vercel 함수 1개(`api/feedback.py`)만 신설(현재 5개 라우트 → 6개, Hobby 12개 제한 내 여유).
- **익명 기본, PII 미수집** — 닉네임은 선택 입력. IP는 요율 제한용 해시로만 쓰고 저장하지 않는다. 양식에 "개인정보·계좌번호 등을 적지 말라"는 안내 문구 고정.
- **데모 접근키 게이트 뒤에 있다** — `DEMO_ACCESS_KEY`가 이미 전 API를 막고 있으므로 공개 스팸 위험은 낮다. 그래도 요율 제한·honeypot·길이 제한은 기본 탑재(키 공유 범위가 넓어질 수 있음).
- **조회는 전원 공개, 상태 변경은 관리자만** — 테스터가 "이미 등록된 건인지" 확인하고 공감(+1)할 수 있어야 중복 등록이 줄고 참여감이 생긴다. 상태(접수→확인중→반영/보류) 변경만 별도 키.

## 3. 데이터 모델 (Upstash Redis)

```
fb:seq                  INCR — 피드백 id 발급
fb:item:{id}            STRING(JSON) — 피드백 본문 (TTL 없음)
fb:index                ZSET — score=등록시각(epoch), member=id  → 최신순 페이징
fb:votes:{id}           INCR — 공감 수
fb:rate:{sha1(ip)}:{d}  INCR+EXPIRE 1d — 등록 요율 제한(기본 10건/일/IP)
```

`fb:item:{id}` JSON 스키마:

```jsonc
{
  "id": 42,
  "ts": "2026-07-08T14:30:00+09:00",
  "type": "quality",            // bug|quality|outdated|missing|idea
  "content": "…",               // 5~1000자, 필수
  "nick": "김PB",               // 선택, ≤20자
  "status": "open",             // open|ack|done|hold  (관리자만 변경)
  "ctx": {                      // 답변 말풍선에서 연 경우 자동 첨부(선택)
    "q": "비밀번호 오류 해제 방법은?",
    "backend": "openai:gpt-4o-mini",
    "gate": {"best": 0.7421, "tau": 0.70, "all_low": false},
    "scope": "화면>계좌",
    "hits": [{"sid": "AC250400", "no": "2504", "path": "계좌 > 고객관리 > …"}]  // 상위 3개
  },
  "ua": "Mobile|Desktop"        // 재현용 최소 구분만(전체 UA 문자열 저장 안 함)
}
```

용량 감각: 피드백 1건 ≈ 1KB. 수백 건 수준이라 Redis 무료 티어(256MB)·명령 쿼터에 영향 없음.

## 4. API — `deploy/online/api/feedback.py` 1개 파일

기존 핸들러 패턴(`suggest.py` 참조: `BaseHTTPRequestHandler` + `_common`의 `authorized`/`send_json`/`_post`) 그대로 따른다.

| 메서드 | 동작 | 비고 |
|---|---|---|
| `POST /api/feedback` | 등록. body=위 스키마의 사용자 입력부 | 검증: type 화이트리스트, content 5~1000자, nick ≤20자, honeypot 필드(`website`)가 비어있을 것, IP 해시 요율 제한. 성공 시 `{id}` 반환 |
| `GET /api/feedback?offset=0&n=20&type=quality` | 최신순 목록 | `ZRANGE fb:index REV` → `MGET fb:item:*` 파이프라인. 유형 필터는 앱단 필터(건수 적음) |
| `POST /api/feedback?action=vote&id=42` | 공감 +1 | 클라이언트 localStorage로 중복 클릭 억제(엄밀할 필요 없음) |
| `POST /api/feedback?action=status&id=42&to=done` | 상태 변경 | `x-admin-key == FEEDBACK_ADMIN_KEY`(신규 env) 필수. 401 그 외 |
| `GET /api/feedback?action=stats` | 통계 집계 | 유형·상태 분포, 최근 14일 일별 등록 추이, 👍/👎 일별 카운터, 공감 상위 5건. 서버 집계 후 단일 JSON 반환(차트는 프런트 렌더) |

👍/👎 원클릭 반응은 목록에 넣지 않고 일별 카운터로만 적재(`fb:up:{yyyymmdd}` / `fb:down:{yyyymmdd}`, 40일 TTL — 기존 `demo:chip:*` 패턴과 동일). 통계의 "답변 만족도 추이" 재료가 된다.

Redis 장애 시: 등록은 503 + "잠시 후 다시 시도" 안내(조용히 삼키지 않음 — 계측 카운터와 달리 사용자 데이터이므로), 조회는 빈 목록 + 오류 배너.

## 5. UI (`public/index.html`·`app.js`·`styles.css`)

진입점 3곳:

1. **답변 말풍선 하단 액션바** — 👍 / 👎 / 「피드백 남기기」. 👎 또는 피드백 클릭 시 모달이 열리고 해당 턴의 `ctx`(질문·근거·게이트)가 자동 첨부된 상태로 시작. 👍는 원클릭 계측(별도 양식 없이 `type=quality, content="👍"` 축약 저장 대신 **Redis 카운터만** — 목록을 오염시키지 않음).
2. **헤더 우측 「피드백」 버튼** — 컨텍스트 없는 전역 등록(버그·제안 등) + 조회 보드 열기.
3. **게이트 차단 턴**("매뉴얼에서 확인되지 않습니다") — "이 내용이 매뉴얼에 있어야 한다고 생각하시면 → 보강 요청" 인라인 링크로 `type=missing` 프리셋 모달. **매뉴얼 공백을 가장 정확히 아는 순간이 바로 이 턴**이므로 놓치지 않는다.

**등록 모달** (기존 scope-panel 모달 톤 유지, Pretendard·현행 디자인 시스템):

```
┌─ 피드백 남기기 ──────────────────────── ✕ ┐
│ 유형   [버그] [답변 품질] [매뉴얼 최신화]      │
│        [매뉴얼 보강] [제안/기타]   ← 칩 선택   │
│ ┌ 💡 이렇게 적어주시면 반영이 빨라요 ────────┐ │
│ │ 어떤 질문에서 · 무엇이 이상했는지 ·         │ │
│ │ 기대한 내용은 무엇인지                     │ │
│ │ 예) "AC250400 비밀번호 해제를 물었는데     │ │
│ │     다른 화면 절차가 답으로 나왔어요"       │ │
│ └──────────────────── (유형별로 교체) ──┘ │
│ 내용   ┌────────────────────────────┐   │
│        │ (placeholder도 유형별 예시)    │   │
│        └──────────────── 5~1000자 ──┘   │
│ 닉네임 [선택, 20자]                        │
│ ▸ 연결된 질문: "비밀번호 오류 해제…" (자동 첨부) │
│   [첨부 제외]                              │
│ ⚠ 개인정보·실계좌정보는 적지 마세요            │
│              [취소]  [등록하기]             │
└──────────────────────────────────────┘
```

**작성 가이드라인 — 유형 칩 선택에 따라 가이드 박스·placeholder가 즉시 교체된다.** 긴 작성 요령 문서를 따로 두지 않고, 입력하는 그 자리에서 "무엇을 적어야 조치 가능한 피드백이 되는지"를 예시로 보여주는 방식(빈 화면 압박 제거 → 작성 진입 장벽 최소화). 가이드 문안은 프런트 상수(`FB_GUIDES`)로 관리:

| 유형 | 가이드 요지 | placeholder 예시 |
|---|---|---|
| `bug` | 어떤 동작을 했을 때 · 무엇이 잘못됐는지 · (가능하면) 브라우저/기기 | "범위를 '업무'로 좁히고 질문하면 로딩이 끝나지 않아요 (모바일 크롬)" |
| `quality` | 어떤 질문에 · 어떤 답이 나왔고 · 왜 틀렸다고 보는지 (질문·답은 자동 첨부됨을 안내) | "근거에는 4단계인데 답변은 3단계까지만 요약했어요" |
| `outdated` | 매뉴얼의 어느 화면/절차가 · 현행 업무와 어떻게 다른지 | "AC250400 비밀번호 해제는 올해부터 OTP 인증이 추가됐는데 매뉴얼에 없어요" |
| `missing` | 무엇을 찾으려 했는지 · 어떤 업무 상황에서 필요한지 (게이트 턴에서 열면 질문 자동 첨부) | "휴면계좌 부활 절차를 물었는데 매뉴얼에서 확인되지 않는대요" |
| `idea` | 자유 서술 — 어떤 상황에서 불편/필요했는지 곁들이면 좋음 | "근거 카드에서 해당 화면 매뉴얼 원문으로 바로 이동하고 싶어요" |

추가 장치: ① 내용 입력이 15자 미만이면 등록 버튼 옆에 "조금만 더 구체적으로 적어주시면 반영에 큰 도움이 돼요" 소프트 안내(차단은 5자 기준 유지 — 강요하지 않음) ② 첫 등록 시도 시 1회만 가이드 박스를 펼침 상태로, 이후 접힘 기본(localStorage) ③ 자동 첨부 항목을 모달에서 눈으로 확인 가능하게 표시(무엇이 전송되는지 투명하게).

**조회 보드** (같은 모달의 탭 또는 전환 뷰):

- 카드 목록: 유형 배지(색상 구분) · 상태 배지(접수/확인중/반영/보류) · 날짜 · 닉네임 · 내용 · 연결 질문(있으면 접힌 인용) · 공감 버튼(♥ n).
- 상단 유형 필터 칩 + 「더 보기」 페이징(20건 단위).
- 관리자 모드: URL `?admin=1` + 키 입력 시 카드에 상태 변경 드롭다운 노출(키는 sessionStorage, 서버가 검증).
- 등록 직후 본인 글이 목록 최상단에 보이게 낙관적 갱신.

내용 렌더링은 반드시 `textContent`(기존 app.js escape 패턴)로 — 사용자 입력 HTML 미해석(XSS 차단).

## 6. 통계 시각화 — 피드백 대시보드

조회 보드 상단에 「통계」 탭. `GET /api/feedback?action=stats` 단일 호출로 집계 JSON을 받아 **프런트에서 인라인 SVG/CSS로 렌더** — 차트 라이브러리(CDN 포함) 불사용. 현행 데모의 무의존 원칙 유지 + 데이터 규모(수백 건)에 라이브러리는 과잉. 렌더 함수는 근거 지도(ev-map)에서 쓰는 것과 같은 vanilla DOM 패턴.

```
┌─ 피드백 통계 ────────────────────────────────┐
│  총 47건   미처리 12   반영 completed 18   공감 132   │  ← 스탯 타일 4개
│                                                │
│  유형 분포                 상태 진행              │
│  버그      ████████ 14    ████████░░░░ 반영 38%  │  ← 가로 막대(값 라벨 병기)
│  답변품질   ██████ 11      (open→ack→done 누적바) │
│  최신화    ████ 7                               │
│  보강      ██████ 10                            │
│  제안      ███ 5                                │
│                                                │
│  일별 등록 추이 (14일)       답변 반응 👍/👎 (14일)  │
│  ▁▂▅▃▇▂▁▁▃▅▂▁▄▆  막대     👍 84% ▲ (스파크라인)   │
│                                                │
│  공감 많은 피드백 TOP 5  →  카드 목록으로 점프       │
└────────────────────────────────────────────┘
```

구성 요소와 목적:

| 위젯 | 형태 | 답하는 질문 |
|---|---|---|
| 스탯 타일 | 숫자 4개(총 건수·미처리·반영·공감 합) | 지금 피드백이 얼마나 쌓였고 얼마나 처리됐나 |
| 유형 분포 | 가로 막대 + 건수 라벨 | 어떤 종류의 문제가 가장 많나 (매뉴얼 문제 vs 데모 품질 문제) |
| 상태 진행 | 단일 누적 바(open/ack/done/hold 색 구분) | 등록한 피드백이 실제로 처리되고 있나 — 테스터 신뢰의 근거 |
| 일별 등록 추이 | 14일 세로 미니 막대 | 테스트 참여가 살아 있나, 공지/배포 이후 반응 |
| 답변 반응 | 👍 비율 % + 일별 스파크라인 | 답변 품질 체감이 개선되고 있나 (양식 미작성 다수 사용자 포함) |
| 공감 TOP 5 | 요약 카드(클릭 → 목록 해당 건) | 여러 사람이 겪는 문제가 무엇인가 — 우선순위 신호 |

색상·형태 규칙: 유형 배지 색을 차트 막대 색과 1:1로 일치시켜 목록↔통계를 오가도 같은 색이 같은 의미(구현 시 dataviz 스킬 가이드 적용). 값은 항상 막대 옆 숫자 라벨 병기 — 툴팁 없이도 읽히게(모바일 우선). 다크 단일 테마(현행 데모와 동일).

집계는 서버(`action=stats`)에서: 목록 전량(`MGET`)을 읽어 유형·상태·일별을 세고, 카운터 키(`fb:up/down:*`)를 14일치 `MGET`. 수백 건 규모라 요청당 집계로 충분하며 별도 집계 저장소 불필요. 관리자 여부와 무관하게 전원 열람 가능.

## 7. 단계별 계획

### 1단계 — 등록 파이프라인 (백엔드 + 모달) · 구현 위임: Opus

1. `api/feedback.py` POST/GET/vote/status/stats + Redis 스키마 (~200줄).
2. 프런트: 답변 액션바(👍/👎/피드백), 유형별 가이드 내장 등록 모달(`FB_GUIDES`), 게이트 턴 보강 요청 링크.
3. `FEEDBACK_ADMIN_KEY` env 추가(Vercel + `.env.local`), README.md(deploy/online) 갱신.

### 2단계 — 조회 보드 + 통계 대시보드 + 관리자 · 구현 위임: Opus

1. 조회 보드 UI(목록·필터·페이징·공감), 헤더 진입 버튼.
2. 통계 탭(§6 위젯 전체 — 스탯 타일·유형/상태 차트·추이·TOP 5, 인라인 SVG).
3. 관리자 상태 변경(서버 검증 + UI 드롭다운).

### 3단계 — 운영 루프 연계 (수집 → 개선 환류)

1. **`deploy/online/export_feedback.py`** (로컬 실행 도구, 배포 제외) — Redis 전량을 `data/feedback.jsonl`로 내려받기. `quality` 건의 (질문, 근거, 게이트) 컨텍스트는 `eval_scope.py` 패턴의 회귀 평가 시드로, `missing`/`outdated` 건은 매뉴얼 보강 요청 목록(마크다운 표)으로 변환.
2. 주간 리뷰 리듬: export → 유형별 분류 → 반영 시 상태 `done` 처리 → 통계 탭의 "반영률"이 올라가는 것을 테스터가 직접 확인(참여 지속 동기).

## 8. 검증 방법

- **API**: `curl`로 ① POST 정상 등록 → id 반환 ② content 4자/1001자 → 400 ③ honeypot 채움 → 400(무음 처리) ④ 11번째 등록 → 429 ⑤ status 변경 무키 → 401, 유키 → 200 ⑥ GET 페이징·최신순 확인 ⑦ `action=stats` — 등록 직후 유형·일별 카운트에 반영 확인.
- **프런트**: 답변 턴에서 피드백 → 모달에 질문 자동 첨부 확인 → 유형 칩 전환 시 가이드/placeholder 교체 확인 → 등록 → 보드 최상단 노출. 게이트 턴에서 보강 요청 프리셋 확인. `<script>` 포함 내용 등록 → 보드·통계 TOP 5 모두 텍스트로만 표시(XSS).
- **통계**: 0건 상태(빈 대시보드 안내), 1건, 수십 건 각각에서 차트 렌더 확인. 모바일 뷰포트에서 라벨 겹침 없는지 확인.
- **회귀**: 기존 5개 API·검색/답변 흐름 무영향(순수 추가 — 기존 파일 수정은 app.js/index.html/styles.css의 UI 추가분만).
- 배포 후 프로덕션에서 위 시나리오 재확인 + Redis 대시보드에서 `fb:*` 키 생성 확인.

| 항목 | 평가 |
|---|---|
| Redis 명령 수 | 등록 4~5 cmd, 조회 2 cmd — 무료 쿼터(500K/월) 대비 무시 가능 |
