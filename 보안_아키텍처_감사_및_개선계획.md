## 0. 문서 목적과 의사결정 요약

이 문서는 다음 세 가지를 하나의 실행 기준으로 통합한다.

1. 현재 코드와 배포 구조에서 **실제로 확인된 취약점과 회귀**를 파일·라인 근거로 기록한다.
2. 공개 합성 데모와 사내 실데이터 서비스의 신뢰 경계를 다시 설계한다.
3. 개선 작업을 우선순위, 의존성, 산출물, 수용 기준, 회귀 테스트가 있는 작업 패키지로 정의한다.

### 핵심 결론

- 파서 계층 복원, 브레드크럼 청킹, 스코프 모델, 근거 카드, 질문뱅크라는 제품 핵심은 유지한다.
- `Claude Code CLI`를 웹 요청의 답변 엔진으로 자동 실행하는 경로는 제거한다.
- 공개 데모는 **승인된 합성 데이터만** 처리하도록 빌드 시점에 강제한다.
- 실제 매뉴얼·상담 데이터는 사내 인증, 로컬 검색, 로컬 LLM, 외부 egress 차단이 적용된 별도 서비스로 분리한다.
- `/api/search`와 `/api/answer`의 중복 검색을 단일 질의 파이프라인으로 통합한다.
- 현재 자기검색 중심 지표를 독립 평가셋 기반의 검색·게이트·인용·최신성 평가로 교체한다.
- 정책 문구를 신뢰하지 않고 `classification`, 데이터 해시, 인증 설정, DLP 결과를 CI와 런타임 불변식으로 강제한다.

### 즉시 중단 대상

## 3. 시스템 이해

### 3.1 제품 목적

PowerBASE 화면·업무 매뉴얼을 구조화하고 로컬 RAG로 검색해, 사내 담당자가 화면 번호·업무 절차·용어를
근거와 함께 확인하도록 하는 시스템이다. 원래의 핵심 제약은 다음과 같다.

- 사내 데이터는 폐쇄망 안에서 처리한다.
- 임베딩, 검색, 리랭커, LLM은 로컬 오픈소스 구성으로 동작한다.
- 답변은 검색된 근거만 사용하고 `[S#]` 인용으로 검증 가능해야 한다.
- 관련도 미달 질의는 생성 모델을 호출하지 않는다.
- 공개 데모는 실제 사내 원문과 분리된 합성 데이터만 사용한다.

### 3.2 현재 데이터 자산

| 데이터 | 규모 | 현재 용도 | 요구 등급 |
|---|---:|---|---|
| 합성 데모 청크 | 832 | 공개 기능 시연 | `PUBLIC_SYNTHETIC` |
| 화면매뉴얼 | 4,108화면 / 39,240청크 | 사내 검색, 현재 온라인에도 반영 | `INTERNAL_CONFIDENTIAL` |
| 업무매뉴얼 | 390문서 / 5,100청크 | 사내 절차 검색, 현재 온라인에도 반영 | `INTERNAL_CONFIDENTIAL` |
| 상담매뉴얼 | 19업무 / 9,123청크 | 상담 Q&A, 온라인 추가 작업 | `RESTRICTED_CASE_DATA` |
| 질문뱅크 | 로컬 1,039 / 온라인 파일 881 수준 | 추천·후속질문 | 원천 데이터 등급 상속 |
| 피드백 | 자유 입력·질문 컨텍스트·캡처 | 품질 개선 | 최소 `INTERNAL_USER_CONTENT` |
| 접근키·관리자키·API 토큰 | 소수 | 인증·외부 서비스 호출 | `SECRET` |

질문뱅크와 파생 메타데이터는 원문을 그대로 포함하지 않더라도 내부 화면명, 질문, 업무 구조를 노출하므로
자동으로 공개 등급이 되지 않는다. **파생 데이터는 원천 데이터의 등급을 기본 상속**해야 한다.

#### 등급별 처리 허용표

| 등급 | 공개 Git | 공개 Vercel | 외부 Vector/LLM | 사내 로컬 저장 | 브라우저 지속 저장 | 기본 보존 |
|---|---|---|---|---|---|---|
| `PUBLIC_SYNTHETIC` | 허용 | 허용 | 승인된 공개 계정만 | 허용 | 최소화 | 데이터 소유자 정책 |
| `INTERNAL_CONFIDENTIAL` | 금지 | 금지 | 원칙적 금지 | 암호화·접근통제 하 허용 | 기본 금지 | 업무상 최소기간 |
| `RESTRICTED_CASE_DATA` | 금지 | 금지 | 금지 | SSO/RBAC·감사 하 제한 | 금지 | 승인된 단기 보존 |
| `INTERNAL_USER_CONTENT` | 금지 | 원본 공개 금지 | 분석 목적 별도 승인 | 권한 저장소만 | 사용자 opt-in·만료 | 예: 본문 90일 |
| `SECRET` | 금지 | 환경 secret만 | 해당 처리자에 필요한 최소값 | secret manager | 금지 | 회전 정책 |

`허용`은 무조건 허용이 아니라 owner, purpose, retention이 manifest에 기록된 경우를 뜻한다. 등급을 알 수 없는 데이터는
가장 높은 제한 등급으로 처리하고 분류 전에는 배포하지 않는다.

### 3.3 로컬 데이터 흐름

```text
사내 RoboHelp/이미지
        |
        v
crawl -> parse -> chunks.jsonl -> FAISS + BM25 + meta
                                      |
브라우저 --HTTP--> webapp.py --------+
                         |            |
                         |            +-> 로컬 리랭커
                         +--------------> Ollama 또는 추출형 답변
                         X--------------> Claude Code CLI (제거 대상)
```

주요 신뢰 경계:

- 브라우저 질의는 신뢰할 수 없는 입력이다.
- 크롤링 원문도 서버 변조·문서 내 지시문을 고려하면 LLM 관점에서는 비신뢰 컨텍스트다.
- FAISS/BM25 아티팩트는 실행 환경에 주입되는 파일이며 무결성 검증이 필요하다.
- Ollama는 생성 전용 서비스여야 하며 파일·명령 도구 권한을 가져서는 안 된다.

### 3.4 현재 온라인 데이터 흐름

```text
브라우저
   |  GET q=질문 + 공유 접근키 헤더
   v
Vercel Python Functions
   |----> Upstash Vector: 질문 임베딩/검색, 실제 hit 전문 저장
   |----> OpenAI: 질문 + 검색 hit 전문 전송
   |----> Upstash Redis: 사용량, 질문추천 계측, 피드백, 캡처
   v
브라우저 localStorage: 접근키 + 최대 30세션 질문/답변/근거 전문
```

현재 구조에서 접근키는 데이터 분류, 전송 승인, 사용자 식별, 문서별 권한, 보존·삭제 정책을 대체하고 있다.
이는 인증 장치가 하나 있다는 뜻일 뿐, 실데이터 처리 경계가 충분하다는 뜻이 아니다.

### 3.5 반드시 코드로 강제할 불변식

| ID | 불변식 | 현재 상태 |
|---|---|---|

| INV-02 | 실데이터 서비스는 인증 설정 누락 시 기동 실패 | 위반(fail-open) |
| INV-03 | 사내 답변 엔진은 외부 egress와 도구 실행 권한이 없음 | 위반 가능 |
| INV-04 | 한 사용자 질문은 검색 1회와 생성 최대 1회만 수행 | 위반 |
| INV-05 | 답변과 근거는 같은 `retrieval_id`와 데이터 버전을 사용 | 미구현 |
| INV-06 | 모든 공개·외부 전송 데이터는 DLP와 승인 manifest를 통과 | 미구현 |
| INV-07 | 점수 필드의 의미는 서버·클라이언트·평가 코드에서 동일 | 위반 |
| INV-08 | 스코프는 매뉴얼·부문 세그먼트 경계를 보존 | 일부 위반 |
| INV-09 | 인덱스·모델·파서·게이트 버전 불일치 시 기동 실패 | 미구현 |
| INV-10 | 사용자 입력·질문·접근키는 URL과 지속 Web Storage에 남지 않음 | 위반 |

### 3.6 보존할 기존 통제와 설계 자산

전면 재작성보다 다음 요소를 보존하고 경계를 강화하는 편이 위험과 비용이 낮다.

- `parse.py`와 `parse_pm.py`의 템플릿 분리, 브레드크럼 계층 복원, 파서 골든 테스트
- `manual`, `sector`, `sector_path`, `section_path`로 이어지는 검색·출처 단위
- 관련도 전건 미달 시 생성 모델을 호출하지 않는 gate 개념
- 답변 `[S#]`과 근거 카드·근거 지도 연결 UX
- 로컬 폰트·무 CDN 구성과 오프라인 모델 경로
- 정적 파일 서빙의 확장자 allowlist와 `resolve()/relative_to()` 경로 탈출 방어
- 일반 답변·피드백 텍스트를 먼저 escape하는 클라이언트 렌더링 원칙
- 질문뱅크를 자기검색으로 사전 검증하는 아이디어. 단, 독립 검증으로 보완

이 항목은 현재 구현 전체가 안전하다는 뜻이 아니다. 예를 들어 텍스트 escape는 이미지 `innerHTML` 경로를 보호하지 못하고,
gate 개념은 caller가 τ를 바꿀 수 있으면 비용·안전 통제가 되지 않는다.

## 6. P0 상세 분석과 즉시 조치

### 6.1 SEC-01: 피드백 이미지 저장형 XSS

**확인 사실**

- `deploy/online/api/feedback.py:143-154`는 문자열 길이와 `data:image/...` 접두만 확인한다.
- Base64 decode, 이미지 magic byte, 실제 디코딩, 안전한 재인코딩을 하지 않는다.
- `deploy/online/public/app.js:1333-1336`은 저장값을 `<img src="${s}">` 문자열로 만들어 `innerHTML`에 넣는다.
- 따옴표와 이벤트 속성이 포함된 조작 문자열이 `_clean_images()`를 그대로 통과하는 것을 재현했다.
- CSP가 없어 inline event handler를 차단할 2차 방어도 없다.

**악용 시나리오**

1. 공격자가 피드백 이미지 필드에 조작된 data URL을 등록한다.
2. 일반 사용자나 관리자가 보드에서 "캡처 보기"를 선택한다.
3. 브라우저가 주입된 이벤트 핸들러를 실행한다.
4. `localStorage`의 데모 접근키, `sessionStorage`의 관리자 키, 저장된 질문·근거를 탈취하거나 관리자 상태 변경을 호출한다.

**즉시 조치**

- `FEEDBACK_IMAGES_ENABLED=0`과 같은 서버측 기능 플래그로 등록·조회 라우트를 모두 비활성화한다.
- 기존 `fb:img:*` 값을 별도 격리하고, 원본을 브라우저에서 열지 않은 채 문자열·Base64 무결성 검사를 수행한다.
- 데모 접근키와 피드백 관리자키를 회전하고 기존 브라우저 세션을 무효화한다.

**영구 수정**

- 서버에서 strict Base64 decode -> 파일 크기 제한 -> 이미지 라이브러리 decode -> 새 이미지로 재인코딩한다.
- 이미지 메타데이터를 제거하고 허용 MIME과 실제 포맷 일치를 검사한다.
- 대형 data URL을 Redis 본문에 보관하지 않고, 짧은 보존기간과 악성 파일 검사가 있는 전용 저장소로 분리한다.
- 프런트는 `createElement('img')`와 DOM property로 `src`를 할당한다. 사용자 데이터를 HTML 문자열 템플릿에 넣지 않는다.
- `script-src 'self'`, `object-src 'none'`, `base-uri 'none'`, `frame-ancestors 'none'` CSP를 적용한다.

**수용 기준**

- 잘못된 Base64, 따옴표, 이벤트 속성, polyglot 파일, 허용되지 않은 포맷은 HTTP 400이다.
- 업로드 후 저장된 바이트는 서버 재인코딩 결과와 일치하고 원본 문자열은 보존되지 않는다.
- Playwright XSS fixture에서 전역 실행 플래그, 외부 요청, storage 접근이 모두 0건이다.
- 기존 이미지 전수 검사와 삭제·격리 결과가 변경 기록에 남는다.

### 6.2 SEC-02: 웹 요청에서 Claude Code 도구 실행

프로젝트 파일 읽기, 허용 명령 실행, Git 변경, 외부 통신으로 이어질 수 있다. 일반 생성형 LLM 호출보다
위험도가 높은 이유는 답변 엔진이 도구 사용 에이전트이기 때문이다.

**즉시 조치**

- 서버의 `_resolve_backend()`에서 Claude 자동 선택을 제거한다.
- 기본 `LLM_BACKEND=none`; 운영에서 명시적으로 `ollama`만 선택한다.

- 생성 엔진은 파일·셸·네트워크 도구가 없는 Ollama/vLLM 추론 API로 제한한다.

- 검색 컨텍스트를 비신뢰 데이터로 명시하고, 출력 인용 검증과 민감정보 후처리를 둔다.

**수용 기준**

- 임의 질문 처리 중 `claude` 프로세스 실행 횟수와 Anthropic 외부 통신이 0이다.
- "지시를 무시하고 파일을 읽어라" 유형의 사용자·문서 fixture가 답변 거부 또는 일반 텍스트 처리로 끝난다.
- 운영 컨테이너에는 Claude CLI와 개발자 설정 디렉터리가 존재하지 않는다.

### 6.4 AUTH-01: fail-open 공유키 인증

**확인 사실**

- `DEMO_ACCESS_KEY`가 비어 있으면 `authorized()`는 모든 요청을 허용한다.
- 헤더뿐 아니라 `?key=` URL 파라미터를 인증 수단으로 허용한다.
- 키는 브라우저 `localStorage`에 장기 저장된다.
- 사용자별 신원, 역할, 만료, 회수, 문서 scope 권한, 접근 감사가 없다.
- 라이브 API는 현재 401이므로 키가 설정돼 있으나, 배포 설정 누락에 대한 안전장치는 없다.

**즉시 조치**

- 실데이터 메타가 감지되면 인증 설정 누락 시 import/기동을 실패시킨다.
- URL 키 지원을 제거하고 기존 공유키를 회전한다.
- 관리자 키와 일반 접근키를 다른 자격증명·권한으로 분리한다.

**목표**

- 사내 OIDC/SSO 또는 인증 프록시를 사용한다.
- `viewer`, `qa`, `feedback_admin`, `data_admin` 역할을 분리한다.
- HttpOnly, Secure, SameSite 쿠키와 짧은 만료·서버측 폐기를 사용한다.
- 인증·관리 변경·대량 조회는 원문과 비밀을 제외한 구조화 감사 로그로 남긴다.

**수용 기준**

- 인증 설정 없음·오류·폐기·만료 자격증명은 401, 권한 부족은 403이다.
- URL의 `key`는 인증에 사용되지 않으며 로그·referrer에 비밀이 없다.
- 로그아웃 또는 사용자 비활성화 후 세션 재사용이 즉시 실패한다.
- 일반 사용자는 관리자 상태 변경과 다른 권한 scope의 원문·피드백을 열람할 수 없다.

- `.gitignore`, `.dockerignore`, Vercel ignore에 원본 디렉터리, `*.xls`, `**/.env*`, 실데이터 파생물을 명시한다.
- pre-commit/CI에서 staged 파일과 Docker context를 검사한다.
- 공개 Git 이력에 포함된 실제 메타데이터의 기밀성을 데이터 소유자가 판단한다. 기밀이면 저장소 비공개 전환과 이력 제거를 별도 승인 절차로 수행한다.

**영구 수정**

- 정규식 하나가 아니라 정책 버전이 있는 DLP 단계와 격리 큐를 둔다.
- 탐지 범위: 전화, 이메일, 계좌, 카드, 주민번호형, 여권/사업자번호형, OTP/장비 일련번호, IP/호스트, 개인명+연락정보 결합.
- 테스트 예외 계좌도 자동 통과시키지 말고 명시된 fixture hash로만 예외 처리한다.
- 청크에 `source_file_hash`, `sheet`, `row`, `effective_at`, `owner`, `classification`, `approval_id`를 저장한다.

**수용 기준**

- high-confidence DLP 검출은 0이거나 만료일 있는 승인 격리 목록에만 존재한다.
- `git check-ignore`와 Docker context 검사에서 원본·비밀·실데이터 파생물이 모두 제외된다.
- 각 상담 답변 근거를 승인된 원본 행과 기준일로 역추적할 수 있다.

## 7. P1 구조·신뢰성 개선

### 7.1 API-01: 중복 검색 제거

현재 프런트는 근거를 먼저 보여주기 위해 `/api/search`를 호출한 뒤 `/api/answer`를 호출한다.
그러나 `/api/answer`가 검색을 다시 수행한다. 온라인 `search()`는 hybrid와 dense gate 요청 두 개를 병렬 호출하므로
질문 하나당 Vector 요청이 기본 4회다. 재시도까지 겹치면 비용과 불일치가 더 커진다.

**개선안**

- 단일 `POST /api/query`가 검색을 정확히 한 번 수행한다.
- SSE `evidence`, `answer`, `done` 이벤트로 근거 선노출 UX를 유지한다.
- SSE가 어려운 환경은 짧은 TTL의 서명된 `retrieval_id`를 반환하고 `/api/answer`가 이를 재사용한다.
- `client_request_id`와 idempotency 저장으로 네트워크 재시도 시 생성 중복을 막는다.
- 브라우저 `AbortController`, 단계별 timeout, `finally` 상태 복구를 구현한다.

**수용 기준**

- 질문 1건당 검색 1회, LLM 생성 최대 1회다.
- 근거와 답변은 동일한 `retrieval_id`, `data_version`, hit 순서를 사용한다.
- timeout·취소·재시도 후 입력 UI가 잠기지 않는다.

### 7.2 API-02: 검색 점수 계약 통일

온라인 게이트는 dense 절대점수를 사용하지만 hit의 `confidence`는 fusion top 대비 상대값이다.
클라이언트는 상대 `confidence`를 τ와 다시 비교해 `low`를 계산한다. 온라인 API는 `alpha`, `rerank`를
파싱하지 않으면서 QA 패널은 이 제어가 적용되는 것처럼 표시한다. `dense`에는 hybrid 상대값, `sparse`에는 0이 들어간다.

**표준 필드**

| 필드 | 의미 |
|---|---|
| `rank_score` | 최종 정렬에 사용한 검색 점수. 검색 방식마다 범위가 다를 수 있음 |
| `dense_score` | 정규화 전 dense cosine 또는 명시된 모델 점수 |
| `sparse_score` | BM25 원점수 또는 표시용 정규화 점수. 둘을 구분 |
| `rerank_score` | cross-encoder 점수. 미사용 시 null |
| `gate_score` | 게이트가 실제 비교한 단일 점수 |
| `gate_threshold` | 서버가 적용한 버전형 임계값 |
| `low_conf` | 서버 판정값. 클라이언트 재계산 금지 |
| `capabilities` | `alpha_tuning`, `rerank_toggle` 등 해당 backend 지원 기능 |

**수용 기준**

- 로컬 webapp, CLI, 온라인 adapter가 같은 fixture에서 동일한 응답 스키마를 반환한다.
- 지원하지 않는 QA 제어는 숨기거나 disabled 사유를 표시한다.
- UI는 `low_conf`를 그대로 사용하고 τ와 임의 비교하지 않는다.

### 7.3 API-03: 입력·비용·가용성 방어

**현재 문제**

- 로컬은 `topk`, `alpha`, `tau`, query 길이에 실질 상한이 없다.
- 온라인은 `tau=nan`이 허용돼 비교 결과가 모두 false가 되면서 게이트를 우회한다.
- caller가 낮은 τ를 지정해 무관 질의도 OpenAI로 보낼 수 있다.
- 로컬 `ThreadingHTTPServer`는 요청마다 무제한 스레드·모델 추론·서브프로세스를 만들 수 있다.
- Redis 사용량 가드는 미설정·장애 시 fail-open이다.

**개선안**

- JSON Schema/Pydantic 등 구조 검증을 사용하고 모든 숫자에 `isfinite`와 범위를 적용한다.
- 일반 사용자 API에서 `tau`, `alpha`, `rerank`를 제거한다. QA 역할의 별도 endpoint만 조정 가능하다.
- 초기 제안: query 1~500자, top-k 1~10, scope 8세그먼트 이하·세그먼트 80자 이하, types enum.
- 사용자·IP·전역 token bucket, 동시 추론 semaphore, request queue 상한을 적용한다.
- 실데이터 생성 비용 가드는 저장소 장애 시 생성만 fail-closed 하고 검색형 폴백을 제공한다.
- 질문은 URL이 아니라 POST body로 받고 프록시·APM 로그에서 body를 redaction한다.

### 7.4 UI-01 / REC-01: 스코프와 추천 회귀

**확인된 회귀**

- 저장된 `상담` scope는 새로고침 시 구형 scope로 오인돼 `화면>상담`으로 변형된다.
- 질문뱅크에 상담 항목이 없을 때 전역 질문으로 폴백하고 상담 scope를 유지해 무관 추천을 만든다.
- 관련질문 인덱스 키가 `sector`뿐이라 화면 `계좌`와 상담 `계좌`가 섞인다.
- 추천 요청 중 scope가 바뀌면 `suggestBusy`가 새 요청을 버리고 이전 응답이 새 scope에 표시될 수 있다.
- 온라인 scope filter는 세그먼트 경계가 아닌 `GLOB 'key*'`를 사용해 wildcard와 접두 충돌 여지가 있다.

**개선안**

- localStorage schema에 명시적 `version`을 두고 유효 루트는 `/api/meta.capabilities.manuals`에서 받는다.
- 질문 인덱스 키를 `(manual, sector, screen_id)`로 구성한다.
- 추천 응답에 추천 자체의 `scope`와 `source_kind`를 넣고 클릭 시 해당 scope를 명시한다.
- active scope에서 후보가 없으면 전역 폴백 대신 "전체 범위에서 보기" 명시적 CTA를 보여준다.
- 추천 요청은 `AbortController` 또는 generation number로 최신 응답만 반영한다.
- filter는 `scope_key = key OR scope_key GLOB 'key>*'`와 서버측 유효 트리 검증을 사용한다.

**수용 기준**

- `상담>계좌` 선택 -> 저장 -> reload -> 검색 round-trip이 동일하다.
- 모든 추천 질문의 scope가 요청 scope의 세그먼트 접두와 일치한다.
- 화면·업무·상담의 동일 sector 이름이 관련 질문에서 교차하지 않는다.
- A scope 요청보다 B scope 요청이 먼저·나중에 도착하는 두 경우 모두 B만 렌더된다.

### 7.5 FB-01: 피드백 개인정보·무결성·보존

현재 안내는 "담당자가 열람"이라고 쓰지만 목록과 이미지는 모든 데모 사용자가 볼 수 있다. 본문·닉네임·질문 컨텍스트·캡처는
TTL과 삭제 API 없이 저장된다. 공감·반응은 서버측 중복 방지나 대상 존재 확인이 없어 통계 조작이 가능하다.

**개선안**

- 원본 피드백은 기본 비공개로 하고 관리자만 열람한다.
- 공개 보드가 필요하면 명시적 공개 동의와 관리자 정제를 거친 별도 레코드만 게시한다.
- 본문 90일, 이미지 30일 등 데이터 소유자가 승인한 보존기간과 자동 삭제를 적용한다.
- 삭제·격리·법적 보존 상태를 지원하고 백업에서도 삭제 추적이 가능해야 한다.
- 질문·답변 원문 대신 redacted snapshot과 immutable `run_id`, data/model/prompt version을 저장한다.
- vote/react는 인증 actor별 1회, 대상 존재 확인, rate limit을 적용한다.
- export는 원문 그대로 저장하지 않고 DLP·Markdown escape·권한 검사를 거친다.

### 7.6 PIPE-01 / PIPE-02: 재현 가능한 데이터 파이프라인

**현재 문제**

- `deploy/build.sh` 기본 수집은 전 부문 manifest가 아니라 계좌 목록 중심으로 동작한다.
- manifest는 Git에서 제외되지만 build가 `crawl_toc.py`를 필수 실행하지 않아 깨끗한 clone의 scope가 비어질 수 있다.
- 일부 크롤 실패와 OCR 미설치를 경고만 하고 전체 build는 성공한다.
- 인덱스 파일을 현재 디렉터리에 순차 기록해 중간 실패 시 혼합 상태가 될 수 있다.
- 코퍼스 변경 후에도 이전 gate τ를 보존하며 재보정을 강제하지 않는다.
- 온라인 resume offset은 두 가변 파일의 연결 순서에 의존하며 삭제된 벡터를 정리하지 않는다.

**개선안**

```text
discover -> fetch -> parse -> validate -> DLP -> chunk -> index -> calibrate -> evaluate -> promote
```

- 각 단계가 manifest와 checksum을 출력하고 다음 단계가 이를 검증한다.
- 기대 토픽 수, 파싱 성공률, 빈 문서 상한, 청크 수 범위, 중복 ID 0을 릴리스 게이트로 둔다.
- 임시 version 디렉터리에서 전체 산출물을 만든 후 atomic symlink/alias로 승격한다.
- content hash 기반 chunk ID와 source별 fingerprint로 변경·삭제 reconciliation을 수행한다.
- 온라인은 새 namespace/index를 완성·평가한 뒤 alias를 전환하고 이전 버전을 롤백용으로 유지한다.
- 메타를 실제 업로드 완료 전에 새 총량으로 공개하지 않는다.

### 7.7 RAG-01: 상담 사례의 최신성·권위

온라인 프롬프트는 상담 사례를 "검증된 정답"으로 보고 원문을 가급적 그대로 옮기게 한다. 그러나 상담 기록은 특정 회원사,
처리 시점, 당시 규정에 종속될 수 있다. 최신 승인 매뉴얼과 충돌할 때의 우선순위도 없다.

**개선안**

- 상담 청크를 `historical_case`로 분류하고 권위 있는 현행 매뉴얼보다 낮은 우선순위를 부여한다.
- `effective_at`, `expires_at`, `member_scope`, `approval_status`가 없는 사례는 생성 답변의 단독 근거로 사용하지 않는다.
- 충돌 시 현행 승인 문서를 우선하고, 해결할 수 없으면 차이를 표시하고 담당자 확인을 요청한다.
- 답변에 회원사 한정·기준일 조건을 보존하되 불필요한 고객사명·개인정보는 노출하지 않는다.

### 7.8 EVAL-01: 자기검색 편향과 품질 증명

기존 `eval_scope.py`는 인덱스 안의 Q&A 질문을 다시 질의해 같은 화면을 찾는 self-retrieval 중심이다.
`calibrate_threshold.py`의 OOD도 고정 15개 수준이며 보정과 보고가 같은 데이터에 의존한다. 이 수치는 파이프라인 회귀에는
도움이 되지만 실제 사용자 답변 정확도를 증명하지 않는다.

**독립 평가셋 구성**

- 실제 사용자 표현을 비식별화한 질문
- 업무 전문가가 작성한 paraphrase, 오타, 약어, 후속질문
- 정답 없음, 도메인 인접 질문, 악성 프롬프트 인젝션
- 화면·업무·상담 충돌과 최신성 충돌
- 정답 화면·경로·핵심 사실·허용 인용·답변 거부 여부를 독립 라벨로 보유
- 보정(train), 개발(dev), 최종 검증(test)을 분리하고 test는 τ 결정에 사용하지 않음

**필수 지표**

| 층 | 지표 |
|---|---|
| 검색 | Recall@1/5, MRR, nDCG@5, manual/sector accuracy |
| 게이트 | AUROC, in-domain FNR, OOD FPR, 근접 도메인 거부율 |
| 답변 | 핵심 사실 정확도, abstention accuracy, 조건 보존율 |
| 인용 | 존재 인용 100%, citation precision/recall, unsupported sentence rate |
| 최신성 | 만료 근거 사용률, 충돌 감지율, 현행 문서 우선율 |
| 운영 | 검색·생성 p50/p95, timeout, fallback 비율, 질문당 Vector/LLM 호출 수 |
| 안전 | PII 출력률, prompt injection 성공률, 권한 밖 hit 반환률 |

## 8. P2 하드닝

### 8.1 보안 헤더와 URL 처리

- Vercel과 로컬 응답에 CSP, `X-Content-Type-Options: nosniff`, `Referrer-Policy: no-referrer`,
  `Permissions-Policy`, `frame-ancestors 'none'`를 적용한다.
- `source_url`은 HTML escape만 하지 말고 URL parser로 `https:`와 허용 host를 검증한다.
- 온라인 실데이터 응답에서는 내부 IP·URL을 제거하고 opaque source ID만 반환한다.
- 클릭재킹, MIME sniffing, 외부 script, inline handler 차단 테스트를 추가한다.

### 8.2 크롤러와 산출물 안전

- topic ID는 매뉴얼별 정규식으로 제한하고 `resolve()/relative_to()`로 출력 경로를 검증한다.
- 응답 최대 크기, 이미지 최대 크기, Content-Type, redirect 대상 host를 제한한다.
- 평문 HTTP 원본은 가능한 경우 인증 프록시와 TLS/mTLS 뒤로 이동한다.
- XLSX 셀 값이 `=`, `+`, `-`, `@`로 시작하면 명시적 문자열로 저장한다.

### 8.3 인덱스 직렬화와 무결성

- `bm25.pkl`을 안전한 구조화 포맷으로 교체하거나, 최소한 manifest 서명·hash 검증 후에만 로드한다.
- 한 바이트라도 변조된 FAISS/BM25/chunks/meta 파일은 서비스 시작 전에 거부한다.
- 인덱스 볼륨은 read-only로 마운트하고 작성자는 별도 빌드 계정으로 분리한다.

### 8.4 공급망과 컨테이너

- 실제 resolver가 만든 hash 포함 lock을 사용하고 Python·모델 revision을 고정한다.
- `curl | sh` 설치는 checksum 검증된 내부 artifact 또는 고정 버전 설치로 교체한다.
- Docker base와 Ollama 이미지를 digest로 고정한다.
- non-root UID, read-only rootfs, `cap_drop: ALL`, `no-new-privileges`, tmpfs를 적용한다.
- SBOM, dependency audit, image scan, secret scan을 CI 필수 체크로 둔다.
- 모델을 이미지에 포함하지 않는다면 기동 전에 캐시 manifest를 검증하고, 깨끗한 호스트에서 실패 원인을 명확히 반환한다.

### 8.5 프런트 접근성·모바일·성능

- 공통 dialog manager로 focus trap, focus restore, `Escape`, background `inert`를 구현한다.
- 이력 항목을 중첩 interactive element가 아닌 올바른 버튼 구조로 바꾼다.
- off-canvas rail/evidence는 닫힐 때 `inert`와 `aria-hidden`을 함께 적용한다.
- tabs에 `aria-selected`, `aria-controls`, roving tabindex를 적용한다.
- 320/375px와 200% 확대에서 헤더, footer action, scope chip, 표가 잘리지 않게 한다.
- 색상 토큰을 WCAG AA 4.5:1 이상으로 보정한다.
- `_static.py`, `_questions.py`, 피드백 모듈의 cold-start 결합을 줄이고 정적 API에 ETag를 적용한다.

#### 프런트 동작 회귀 보완

- 전역 `S.busy` 대신 턴 단위 요청 상태와 취소 기능을 사용한다.
- 검색 성공 후 생성만 실패하면 확보한 근거를 유지하고 "답변만 다시 생성"을 제공한다.
- 401, 403, 429, timeout, offline, backend 장애를 서로 다른 상태·행동으로 안내한다.
- 피드백 API가 `response.ok && body.ok === true`일 때만 성공 처리한다. 실패 시 입력을 지우지 않는다.
- 피드백 취소 시 본문·닉네임·질문 컨텍스트·이미지를 모두 폐기하거나 모두 임시 저장하는 일관된 정책을 사용한다.
- clipboard API가 실패하면 "복사됨"을 표시하지 않는다.
- 자동 스크롤은 사용자가 하단 근처에 있을 때만 수행한다.
- gated/empty/error 문구의 "계좌" 하드코딩을 제거하고 실제 manual/scope를 표시한다.

## 9. 목표 아키텍처

### 9.1 배포 경계

```text
                           +-------------------------------+
Public Internet ---------->| Public Synthetic Demo         |
                           | - synthetic corpus only       |
                           | - separate project/index/key  |
                           | - no internal source metadata |
                           +-------------------------------+

Corporate SSO/VPN -------->| Internal Manual Service       |
                           | - RBAC + audit                 |
                           | - FAISS/BM25 local             |
                           | - Ollama local, tools disabled|
                           | - outbound network denied     |
                           +-------------------------------+

Data Admin --------------->| Offline Build Zone            |
                           | - crawl/parse/DLP/approval     |
                           | - versioned index + promotion  |
                           +-------------------------------+
```

공개 데모와 내부 서비스는 같은 UX 개념과 API schema를 공유할 수 있지만, 데이터 plane과 자격증명은 공유하지 않는다.

### 9.2 애플리케이션 계층

```text
HTTP Adapter
  -> Request Validator / AuthZ / Rate Limit
  -> QueryService
       -> Retriever interface
            -> LocalFaissBm25Retriever
            -> SyntheticUpstashRetriever (public only)
       -> GatePolicy
       -> AnswerGenerator interface
            -> OllamaGenerator
            -> ExtractiveGenerator
            -> PublicDemoGenerator
       -> CitationValidator / DLP Output Filter
  -> SSE/JSON Presenter
```

`src/chatbot.py`, `src/webapp.py`, 온라인 `_common.py`에 복제된 검색·게이트 개념을 이 계약에 맞춰 통합한다.
온라인 serverless 패키지 제약 때문에 구현을 완전히 공유하기 어렵다면, schema와 fixture 기반 contract test는 반드시 공유한다.

### 9.3 표준 질의 API

일반 사용자는 검색 튜닝값을 보내지 않는다.

```http
POST /api/query
Content-Type: application/json
Idempotency-Key: <uuid>

{
  "query": "비밀번호 오류 횟수를 해제하는 화면은?",
  "scope": ["화면", "계좌"],
  "top_k": 5,
  "mode": "standard",
  "client_request_id": "<uuid>"
}
```

SSE 예시:

```text
event: evidence
data: {"retrieval_id":"r_...","data_version":"idx_...","gate":{"score":0.73,"threshold":0.56,"passed":true},"hits":[...]}

event: answer
data: {"retrieval_id":"r_...","answer":"... [S1]","citations":[{"rank":1,"supported":true}],"backend":"ollama"}

event: done
data: {"request_id":"...","search_ms":320,"generation_ms":4120}
```

QA 튜닝은 `qa` 역할만 접근 가능한 별도 endpoint와 별도 UI에서 수행하며, 변경값은 운영 게이트에 자동 반영하지 않는다.

### 9.4 데이터셋 manifest

```json
{
  "dataset_id": "pb-manual-2026-07-10.1",
  "classification": "INTERNAL_CONFIDENTIAL",
  "corpus_sha256": "...",
  "source_manifest_sha256": "...",
  "parser_version": "git:...",
  "chunk_schema": 2,
  "embedding_model": "jhgan/ko-sroberta-multitask@<revision>",
  "reranker_model": "BAAI/bge-reranker-v2-m3@<revision>",
  "gate_policy_version": "gate-2026-07-10.1",
  "dlp_policy_version": "dlp-1",
  "dlp_high_findings": 0,
  "owner": "<role>",
  "approval_id": "<change-ticket>",
  "created_at": "2026-07-10T00:00:00Z"
}
```

서비스는 manifest의 모델·schema·hash와 런타임이 일치하지 않으면 준비 상태를 false로 유지한다.

### 9.5 제안 코드 배치

```text
src/pb_rag/
  contracts/           # Chunk/Search/Gate/Answer/Feedback v2 schema
  retrieval/           # LocalFaissBm25Backend, public synthetic adapter
  generation/          # Ollama/Extractive/PublicDemo adapter, 도구 없음
  policies/             # authz, gate, source authority, citation, DLP output
  service/              # QueryService orchestration

pipelines/
  sources/              # ST/PM/XLS source adapter
  validation/           # 수지, schema, DLP, provenance
  release/              # immutable build, evaluate, promote, rollback

services/
  internal/             # SSO/RBAC, local backend, egress denied
  public/               # synthetic-only policy, 별도 credentials

web/src/
  core/                 # thread, evidence, scope, request state
  adapters/             # internal/public capability 차이
  features/             # feedback, onboarding, QA lazy modules
```

`src/chatbot.py`는 별도 검색 구현을 가지지 않고 공용 QueryService를 호출하는 CLI adapter로 축소한다. 온라인 serverless가
동일 Python 패키지를 쓸 수 없다면 JSON Schema와 fixture contract를 공유하고 생성된 API 모델의 diff를 CI에서 검사한다.

### 9.6 클라이언트 상태 머신

```text
unauthorized
    -> idle
        -> searching
            -> answering
                -> done | gated
            -> error | timeout | cancelled
        -> error | timeout | cancelled
```

상태별 불변식:

| 상태 | 보존 데이터 | 허용 행동 |
|---|---|---|
| `unauthorized` | 민감 데이터 없음 | 로그인·재인증만 |
| `idle` | 사용자 opt-in 이력만 | 질문, scope 변경 |
| `searching` | 질문·scope·request ID | 취소 |
| `answering` | 확정 evidence와 retrieval ID | 취소, 근거 열람 |
| `done` | 답변·검증된 citation | 복사, 피드백, 후속질문 |
| `gated` | 참고 근거와 gate reason | 질문 수정, 범위 변경 |
| `error` | 성공한 이전 단계 결과 | 실패 단계만 재시도 |
| `timeout` | 성공한 이전 단계 결과 | 재시도·취소 |
| `cancelled` | 서버 생성 결과 미반영 | 새 질문 |

DOM 전체 교체가 아니라 상태 노드를 고정하고 필요한 본문만 갱신해 screen reader 공지와 포커스를 보존한다.

### 9.7 오류 계약

모든 API 오류는 HTML이나 임의 문자열이 아니라 다음 구조를 사용한다.

```json
{
  "error": {
    "code": "retrieval_timeout",
    "message": "검색 시간이 초과되었습니다.",
    "retryable": true,
    "request_id": "req_...",
    "details": null
  }
}
```

`message`에는 내부 URL, stack trace, token, 원문 hit를 넣지 않는다. 클라이언트는 `code`와 `retryable`로 행동을 결정한다.

### 9.8 무중단 마이그레이션

1. 현행 동작을 재현하는 characterization test와 알려진 취약점 회귀 테스트를 먼저 만든다.
2. 기존 chunk와 API를 읽는 `V1Adapter`를 두고 v2 schema를 병행한다.
3. 현재 index A를 서비스하면서 release B를 별도 namespace에 전량 구축한다.
4. 실제 트래픽은 답변 생성 없이 retrieval만 shadow 실행해 hit·gate 차이를 기록한다.
5. v2 프런트는 feature flag로 내부 사용자부터 10% -> 50% -> 100% canary 전환한다.
6. index A와 앱 이전 버전은 최소 2개 release 또는 14일 보존하고 alias 전환으로 롤백한다.
7. 두 번의 안정 릴리스 후 legacy endpoint, V1Adapter, 중복 검색 코드를 제거한다.

## 11. 단계별 실행 로드맵

### P0. Containment — 즉시, 0~1일

**목표:** 추가 노출과 즉시 악용 경로를 중단한다.

1. 온라인 실데이터 인제스트와 배포를 동결한다.
2. 피드백 이미지 등록·조회 기능을 서버에서 비활성화한다.
3. Claude 자동 backend를 제거하고 로컬 서비스를 `none`으로 안전 기동한다.
4. 키를 회전하고 URL 키를 제거한다.
5. 원본 XLS·실데이터 파생물의 Git/Docker/Vercel ignore를 보강한다.
6. 외부 저장·전송 범위와 공개 Git 메타데이터를 조사한다.
7. 공개 데모를 승인된 합성 인덱스로 복구한다.

**릴리스 게이트:** P0 항목 중 하나라도 미완료면 온라인 재배포 금지.

### P1. Security Boundary — 2~5일

**목표:** 인증과 데이터 분리를 문서가 아니라 실행 환경에서 강제한다.

1. 공개/내부 프로젝트·키·인덱스·CI를 분리한다.
2. 내부 서비스에 SSO/RBAC와 감사 로그를 적용한다.
3. POST 입력 schema, 유한수/길이/enum 검증, rate limit, 동시성 상한을 구현한다.
4. CSP·보안 헤더·URL allowlist를 적용한다.
5. 피드백 원본을 비공개로 전환하고 보존·삭제·격리 정책을 구현한다.
6. DLP scanner와 배포 전 분류 manifest 검증을 추가한다.

**릴리스 게이트:** 비인증·권한·XSS·DLP·prompt injection 필수 테스트 전부 통과.

### P2. Query Core — 1주

**목표:** 검색·게이트·답변·UI 계약을 하나로 만든다.

1. QueryService와 Retriever/Generator interface를 도입한다.
2. 단일 `/api/query`와 SSE 또는 retrieval ID 흐름을 구현한다.
3. 점수 schema와 `low_conf` 서버 판정을 통일한다.
4. idempotency, timeout, 취소, retry 정책을 적용한다.
5. manual-aware 추천 인덱스와 scope schema v2를 적용한다.
6. 프런트 공통 코어와 local/online adapter를 분리하거나 생성 검증을 둔다.

**릴리스 게이트:** 질문당 검색 1회, backend contract, scope E2E, 중복 생성 테스트 통과.

### P3. Versioned Data & Quality — 1~2주

**목표:** 어떤 데이터·모델·정책으로 답했는지 재현하고 품질을 독립적으로 증명한다.

1. provenance와 classification이 포함된 chunk schema v2를 만든다.
2. 단계별 manifest, content hash ID, DLP, atomic promotion을 구현한다.
3. 독립 train/dev/test 평가셋을 구축한다.
4. 검색·게이트·답변·인용·최신성·안전 지표를 자동 산출한다.
5. 상담 사례의 효력일·회원사 범위·승인상태 정책을 적용한다.

**릴리스 게이트:** 이전 baseline 대비 유의한 회귀 없음, DLP·권한 누수 0, 롤백 훈련 통과.

### P4. Production Hardening — 지속

**목표:** 장애·공급망·접근성·운영 피드백을 지속적으로 관리한다.

1. non-root/read-only 컨테이너, digest, SBOM, 취약점 스캔을 적용한다.
2. request/data/model version과 fallback 사유를 구조화 계측한다.
3. Playwright·axe로 모바일·키보드·접근성을 릴리스마다 검증한다.
4. 키 회전, 복구, 데이터 삭제, 사고 대응 훈련을 정기 실행한다.
5. 위험 예외는 만료일과 책임자가 있는 risk register로 관리한다.

### 의존관계

```text
P0 Containment
  -> P1 Security Boundary
      -> P2 Query Core
          -> P3 Versioned Data & Quality
              -> P4 Production Hardening

P1 DLP ---------------------> P3 Data Promotion
P1 Auth/RBAC ---------------> P2 QA/Admin API
P2 Response Schema ---------> P3 Evaluation Harness
```

### 12.2 API 계약

- 잘못된 JSON, content type, 알 수 없는 필드는 일관된 4xx 문제 응답을 반환한다.
- `retrieval_id`, `data_version`, hit rank가 evidence와 answer에서 동일하다.
- 모든 `[S#]`는 실제 hit 범위 안에 있고 클릭 가능한 출처가 존재한다.
- citation 0개 또는 지원되지 않는 문장이 있으면 생성 답변을 실패·수정·발췌 폴백한다.
- timeout 후 같은 idempotency key 재시도는 기존 결과를 반환하며 LLM을 다시 호출하지 않는다.
- backend별 capability와 score schema가 fixture contract를 만족한다.

### 12.3 데이터 파이프라인

- 깨끗한 clone fixture에서 TOC -> 수집 -> 파싱 -> 청크 -> 색인 -> 보정 -> 평가가 재현된다.
- expected topics, parsed docs, chunks, empty docs, duplicate IDs가 승인 범위와 일치한다.
- 한 원본 수정은 예상한 content hash ID만 변경하고 stale vector는 제거된다.
- 중간 실패 시 현재 운영 index는 변하지 않는다.
- DLP 차단 fixture와 승인 예외 fixture가 정책 버전별로 재현된다.
- 모델 revision 또는 chunk schema가 다르면 서비스 시작이 실패한다.

### 12.4 RAG 품질

- 독립 test set은 인덱스 Q&A 원문과 exact/near duplicate가 아님을 검사한다.
- manual/sector scope leakage는 0이어야 한다.
- 게이트 threshold는 train/dev에서만 결정하고 test는 최종 보고에만 사용한다.
- 답 없음 질문, 근접 도메인, 악성 지시, 최신성 충돌을 별도 cohort로 보고한다.
- metric은 단일 숫자뿐 아니라 bootstrap 신뢰구간과 이전 배포 대비 차이를 기록한다.

### 12.5 프런트 E2E·접근성

- 상담 scope 선택 -> 저장 -> reload -> 추천 -> 질문 흐름이 유지된다.
- scope 요청 응답 순서를 뒤집어도 최신 scope만 렌더된다.
- 검색 timeout, answer timeout, 취소, 재시도 후 입력과 버튼이 복구된다.
- 일반 사용자는 비공개 피드백과 이미지를 열람할 수 없다.
- `Tab`, `Shift+Tab`, `Enter`, `Space`, `Escape`로 모든 핵심 흐름을 완료한다.
- 320px, 375px, 태블릿, 데스크톱, 200% 확대에서 가로 overflow와 잘린 action이 없다.
- axe critical/serious 0, 텍스트 대비 AA 충족, reduced motion 적용을 확인한다.

## 14. P0 사고대응 Runbook

직접 삭제·초기화 명령은 이 문서에 넣지 않는다. 외부 데이터 삭제와 Git 이력 변경은 증적 보존과 승인 후 수행한다.

1. **변경 동결**: 담당자가 배포·인제스트 freeze를 선언하고 작업 티켓을 연다.
2. **증적 보존**: 현재 commit, Vercel deployment ID, Vector count, Redis key count, 환경 설정의 존재 여부만 기록한다. 비밀 값은 기록하지 않는다.
3. **공격면 차단**: 이미지 라우트, 실데이터 answer, Claude backend를 비활성화한다.
4. **자격증명 회전**: Demo/Admin/Upstash/OpenAI를 회전하고 기존 세션을 무효화한다.
5. **범위 조사**: 저장 데이터, API 호출, 접근 로그, OpenAI 전송·보존 범위를 시간순으로 정리한다.
6. **데이터 결정**: 데이터 소유자·보안·개인정보 담당자가 삭제, 격리, 신고, 사용자 통지 필요성을 판단한다.
7. **합성판 복구**: 승인된 합성 corpus hash로 공개 서비스를 재구축한다.
8. **검증**: 무인증 401, `demo=true`, XSS 차단, 내부 식별자 0을 독립 검토자가 확인한다.
9. **사후 분석**: 원인, 탐지 실패, 영향을 기록하고 CI 불변식으로 재발 방지를 구현한다.

## 20. 완료 정의

이 계획은 기능이 추가됐을 때가 아니라 다음 조건이 모두 충족될 때 완료된다.

1. 공개 빌드에 실데이터가 구조적으로 들어갈 수 없다.
2. 사내 질의가 외부 서비스나 도구 실행 에이전트로 나갈 수 없다.
3. 인증·권한·DLP·인덱스 버전이 설정 누락 시 안전하게 실패한다.
4. 한 질문의 근거와 답변이 같은 검색 결과·데이터 버전에서 생성된다.
5. 답변의 모든 인용이 실제 근거를 가리키고 최신성·회원사 조건을 보존한다.
6. 독립 평가셋과 자동 보안·계약·E2E 테스트가 릴리스를 차단할 수 있다.
7. 데이터 저장·전송·삭제와 사용자·관리자 행위를 감사할 수 있다.
8. 장애와 잘못된 인덱스 배포에서 직전 버전으로 검증된 롤백이 가능하다.

이 기준을 통과한 뒤에 질문 추천, 멀티턴, 스트리밍, VLM 고도화 같은 기능 개선을 재개한다.
