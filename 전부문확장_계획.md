# 전 부문 확장 계획서 — 브레드크럼 스코프와 교차 오염 방지

> 계좌(AC*, 356화면/4,443청크)에서 PowerBASE 온라인 매뉴얼 **전 부문**으로 확장한다.
> 핵심 가치: ① 전 부문에서 **동일한 브레드크럼 체계** 제공 ② 사용자가 질문 범위를
> **브레드크럼으로 필터링/선택** ③ 타 부문 검색 결과와 겹칠 때 **시스템이 먼저 알려주고
> 사용자가 범위를 확정하는 피드백 루프**. 셋이 한 몸이다 — 부문이 늘수록 동음이의
> 용어("매수", "약정", "해지")가 폭증하므로, 스코프 없는 확장은 품질 하락이다.

## 0. 문제 정의 — 왜 "그냥 전량 색인"이면 안 되는가

| 확장 시 발생하는 문제 | 예시 | 대응 장치 |
|---|---|---|
| **동음이의 교차 오염** | "약정 해지"가 계좌·신용·대출 부문에서 모두 히트 → 근거가 섞여 답변이 흐려짐 | 부문 스코프 필터 + 모호성 감지 |
| **사용자는 자기 부문만 궁금함** | 계좌 담당자의 "고객정보 변경"에 주문 부문 근거는 소음 | 브레드크럼 스코프 셀렉터(기본 범위 기억) |
| **모호한 질문은 시스템도 모름** | "수수료 조회"는 어느 부문? — 임의로 고르면 틀린다 | **스코프 힌트 피드백**: 부문 분포를 보여주고 사용자가 원클릭 확정 |
| **파서 전제 붕괴 위험** | 타 부문 HTML이 다른 CSS class 템플릿일 수 있음 | 부문별 파일럿 → 골든 테스트 누적(기존 계좌 골든 불변) |

## 1. 현재 구조 진단 (확장 관점)

- `crawl.py` — **TOC 트리 파싱 없음**. 토픽 코드 목록(`account_topics.txt`)을 받아 내려받기만 한다. 부문 정보가 파이프라인 어디에도 없음 → **①번 공사**
- `to_chunks.py` — `section_path`의 루트가 **화면 제목**. 부문 레벨이 없어 "고객정보조회"가 어느 부문 소속인지 청크만 보고 알 수 없음 → **②번 공사**
- `webapp.py search()` — 후보 루프에서 `chunk_type` 필터만 존재. 같은 자리에 경로 접두 필터를 끼우면 됨(구조 재사용) → **③번 공사**
- UI — 근거 지도가 이미 브레드크럼 트리를 렌더하므로, 부문 레벨 확장과 "이 경로로 좁히기" 액션의 토대가 됨

## 2. 핵심 설계

### 2-A. 2계층 브레드크럼 — TOC 경로 + 문서 내 경로

브레드크럼을 **매뉴얼 목차(TOC) 경로**와 **문서 내 경로**의 결합으로 재정의한다.

```
[TOC 경로]              [문서 내 경로 — 기존 section_path]
계좌 › 고객관리     ›    고객정보조회 › 화면설명 › 조건입력 › 종합계좌번호
└ sector_path            └ section_path (불변)
```

**청크 스키마 확장** (기존 필드 불변 — 하위 호환):

```json
{
  "sector":       "계좌",                     // 부문 (1차 스코프 키)
  "sector_path":  ["계좌", "고객관리"],        // TOC 전체 경로
  "scope_key":    "계좌/고객관리/AC110100",    // 필터용 정규화 키 (경로 접두 매칭)
  "embed_text":   "[계좌] 고객정보조회 > 화면설명 > … : …"   // 부문 접두 — 임베딩에도 부문 문맥 주입
}
```

- `embed_text`에 부문을 접두하는 근거: Anthropic Contextual Retrieval 실측 — 청크에 상위 문맥을 보존하면 검색 실패율 35~49%↓. 부문이 곧 최상위 문맥이다.
- **주의**: embed_text 변경은 전 청크 재임베딩을 뜻함 — 어차피 전량 확장 빌드이므로 이번이 유일한 무비용 타이밍.

### 2-B. 브레드크럼 스코프 검색 — `scope` 파라미터

```
GET /api/search?q=…&scope=계좌              # 부문 스코프
GET /api/search?q=…&scope=계좌/고객관리      # TOC 중분류까지
GET /api/search?q=…&scope=계좌//AC110100    # 특정 화면까지 (경로 접두 매칭)
GET /api/sectors                            # 스코프 셀렉터용 TOC 트리 (부문→중분류→화면, 청크 수 포함)
```

- 구현: `search()` 후보 루프의 `chunk_type` 필터 옆에 `scope_key.startswith(scope)` 한 줄 — dense는 전체 검색 후 마스킹(pool을 스코프 내에서 채울 때까지 순회), BM25도 동일. 부문 수가 늘어도 단일 FAISS 인덱스 유지(수만 청크는 Flat으로 충분, 분리 인덱스는 운영 복잡도만 증가)
- 게이트(τ)는 스코프 적용 **후** 후보에 대해 판정 — 스코프를 좁힐수록 "확인되지 않음"이 정직해짐

### 2-C. 모호성 감지와 사용자 피드백 루프 (핵심 가치)

스코프 없이 질문했을 때, 서버가 **근거의 부문 분포**를 계산해 응답에 실어준다:

```json
"scope_hint": {
  "ambiguous": true,                  // 상위 근거가 2개 이상 부문에 분산 + 점수 격차 근소
  "sectors": [
    {"sector": "계좌", "count": 3, "best": 0.71},
    {"sector": "신용", "count": 2, "best": 0.69}
  ]
}
```

판정 규칙(초안): top-k 근거의 부문 수 ≥ 2 **그리고** 1·2위 부문의 best 신뢰도 차 < 0.08 → ambiguous. (임계값은 P4 방식의 분포 보정으로 확정)

**UI 피드백 배너** — 답변 카드 상단에:

```
⚠ 근거가 두 부문에 걸쳐 있어요 — 어느 업무 기준으로 볼까요?
[ 계좌에서만 (3건 · 0.71) ]  [ 신용에서만 (2건 · 0.69) ]  [ 전체 유지 ]
```

- 칩 클릭 → 같은 질문을 해당 스코프로 **즉시 재검색**(재입력 없음). 선택은 대화 스코프로 승격되어 후속 질문에 상속
- ambiguous가 아니어도 근거가 단일 부문이면 조용히 "범위: 계좌" 칩만 표시 — 사용자는 항상 **지금 어느 부문을 보고 있는지** 안다

### 2-D. UI — 스코프가 1급 시민이 되는 세 지점

```
┌─ 컴포저 ──────────────────────────────────────────────┐
│ [범위: 계좌 › 고객관리 ✕]  매뉴얼에 대해 물어보세요   │ ← ① 스코프 칩 (클릭=셀렉터, ✕=전체)
└───────────────────────────────────────────────────────┘
① 스코프 셀렉터: 칩 클릭 → 부문 › 중분류 › 화면 캐스케이딩 패널(/api/sectors, 검색 가능).
   선택은 localStorage에 기억 — 계좌 담당자는 늘 계좌가 기본값
② 모호성 배너: 2-C — 결과 위에서 원클릭 재스코프
③ 근거 지도: 트리 루트에 부문 레벨 추가. 노드 hover 시 [🔍 이 경로로 좁히기]
   액션 → 해당 경로를 스코프로 재검색. "지도가 곧 필터"가 되어
   브레드크럼 탐색(선택)과 검색(질문)이 한 화면에서 순환한다
```

이력·QA 모드·정밀 토글 등 기존 UI는 불변. QA 모드에는 부문별 히트 분포 계측 추가.

## 3. 데이터 파이프라인 확장

### 3-A. TOC 전수 크롤 (신규: `src/crawl_toc.py`)

- RoboHelp 목차(`toc*.js`, whxdata)를 재귀 파싱 → `data/manifest.json`:
  `[{screen_id, title, sector_path:["계좌","고객관리"], url}]` — 브레드크럼 상위 계층의 **단일 원천**
- 부문별 토픽 목록 `data/topics/<부문>.txt` 파생(기존 `--from-file` 재사용, 부문 단위 증분 수집)
- **선행 조사(사내망 필수)**: 전 부문 목록·토픽 수·HTML 템플릿 동질성 샘플링 리포트 — 이 결과로 3-B 파일럿 대상 확정

### 3-B. 부문별 파일럿 → 파서 보강 (계좌 방법론 반복)

- 부문마다 대표 화면 1~2개 → `parse.py` 통과 검증 → **부문별 골든값을 tests/test_parse.py에 누적**
- 회귀 제약: **기존 계좌 골든 5종은 절대 불변** — 파서 보강은 추가 규칙으로만
- Claude Code 실습 프롬프트(교육자료 P4에 이미 수록된 패턴 재사용):
  > "data/html에 ○○ 부문 샘플 3개를 받아왔어. 계좌 파서가 그대로 동작하는지 검증하고, 깨지는 규칙이 있으면 골든값을 추가한 뒤 파서를 보강해줘. 기존 계좌 골든값은 깨지면 안 돼."

### 3-C. 청크·색인 재빌드

- `to_chunks.py`: manifest 조인으로 sector/sector_path/scope_key 부여, embed_text 부문 접두
- `build_index.py`: 변경 불요(스키마 투과). meta.json에 `sectors:{부문:청크수}` 기록
- 규모 추정: 부문 수 × 계좌 규모 가정 시 수만 청크 — Flat 인덱스·CPU 임베딩 모두 허용 범위(빌드 수십 분, 1회성). 초과 시 IVF 전환은 후순위 옵션으로만 명시

## 4. 품질 검증 — "겹치지 않음"을 수치로 증명

1. **교차 오염 평가 세트(신규)**: 부문별 qa 청크에서 (질문, 정답 화면, 정답 부문) 쌍 추출 + **의도적 동음이의 질의 세트**("약정 해지", "수수료", "비밀번호 변경" 등 부문 중복 용어 30개)
2. 지표:
   - 스코프 없음: Recall@5, **부문 정확도**(1위 근거의 부문 = 정답 부문 비율), ambiguous 감지 정밀/재현율
   - 스코프 지정: Recall@5 (동일 질의에서 스코프 지정 시 개선 폭 = 이 기능의 존재 증명 수치)
3. τ **재보정**: 코퍼스가 커지면 신뢰도 분포가 이동 — `calibrate_threshold.py`를 확장 코퍼스에서 재실행(양성 질의를 부문별 균형 샘플로)
4. 기존 계좌 실측 시나리오 4종은 전량 색인 후에도 동일 품질이어야 함(회귀 기준)

## 5. 실행 로드맵

| 단계 | 내용 | 산출/게이트 | 규모 |
|---|---|---|---|
| **P1** | TOC 전수 크롤러 + 부문 매니페스트 (사내망 조사 포함) | manifest.json · 부문/토픽 수 리포트 | 1일 |
| **P2** | 부문별 파일럿 파싱 검증·파서 보강 | 부문별 골든 테스트 전부 통과, 계좌 골든 불변 | 부문당 0.5일 |
| **P3** | 청크 스키마 확장 + 전량 수집·색인 | sectors 통계 포함 meta.json, 전량 인덱스 | 1일+빌드 |
| **P4** | 서버: scope 필터·scope_hint·/api/sectors | curl 계약 테스트(스코프별 결과 분리 확인) | 1일 |
| **P5** | UI: 스코프 칩·셀렉터·모호성 배너·근거 지도 좁히기 | 시나리오 스크린샷 검수 | 2일 |

## 7. 이 계획의 성공 기준 (한 줄씩)

- 사용자는 **항상 지금 어느 부문을 보고 있는지 안다** (스코프 칩 상시 노출)
- 모호한 질문에서 시스템이 **먼저 물어본다** ("계좌 기준? 신용 기준?") — 틀린 답 대신 되묻기
- 동음이의 질의 세트에서 **부문 정확도·Recall@5가 스코프 도입 전보다 정량 개선**
- 계좌 부문의 기존 품질(골든 테스트·실측 시나리오)이 **1도 후퇴하지 않는다**
