# 임원 보고 PDF 제작 계획서 — PB 매뉴얼 데스크 개발 보고

> 완성된 소개 영상(report.mp4/full.mp4)과 함께 배포할 **임원 보고용 프리미엄 PDF**.
> 지금까지의 개발 내용을 마크다운 원고 + 실제 화면·다이어그램 이미지로 구성해
> **애뉴얼리포트급 편집 품질**로 만든다. 영상이 "보여주는" 자료라면, PDF는 "남는" 자료다.

---

## 1. 개요

| 항목 | 내용 |
|---|---|
| 목적 | 직책자·임원이 **5분 안에 훑고, 회의에서 근거로 인용**할 수 있는 개발 성과 보고서 |
| 톤 | 프리미엄·절제 — 화려함보다 **고급스러운 밀도**(여백, 타이포, 정돈된 인포그래픽) |
| 분량 | **A4 세로 8쪽 내외** (표지 1 + 본문 6 + 클로징 1) — 임원 문서는 짧을수록 강하다 |
| 산출물 | `PB매뉴얼데스크_개발보고_2026-07.pdf` + 원고(md)·템플릿·빌드 스크립트(재생산 가능) |
| 배포 | 사내 한정 (매뉴얼 원문 스크린샷 포함). 영상 파일과 같은 폴더로 패키징 |
| 영상 연계 | 본문에 영상 캡처 + **재생 안내 박스**("동봉된 report.mp4 참조, 2분 58초") 삽입 |

## 2. 디자인 방향 — "있어 보이게"의 구체안

영상과 **하나의 브랜드 시스템**으로 통일하되, 인쇄물 문법으로 변환한다:

- **표지**: 영상 인트로와 동일한 다크 네이비(#0b0e13) 풀블리드 + koscom 오렌지 그라데이션 라인 + 스파크 심볼. 큰 타이틀·날짜·부서명만 — 요소 4개 이하
- **본문**: 라이트(화이트) 배경 — 인쇄·가독성 표준. 오렌지(#e4670a)는 **강조 전용**(수치·키워드·섹션 라벨), 본문은 잉크(#191c20)/그레이 2단계
- **타이포**: Pretendard 단일 패밀리로 웨이트 대비(900 헤드라인 / 400 본문 / 600 캡션). 숫자는 크게 — "356" 같은 수치가 페이지의 주인공
- **레이아웃 모티프**: 영상의 씬 라벨("— WHY", "— STATUS")을 **섹션 킥커**로 재사용. 페이지 하단에 얇은 오렌지 풋터 라인 + 페이지 번호
- **인포그래픽**: 표 대신 **스탯 카드·타임라인·파이프라인 다이어그램**. 스크린샷은 반드시 브라우저 프레임/그림자 처리로 "떠 있게"
- **금지**: 클립아트, 이모지 남발, 3색 이상 팔레트, 빽빽한 표 — 싸 보이는 지름길

|---|---|---|---|

| 4 | **솔루션 — 무엇이 다른가** | 세 가지 약속: ① 자연어 질문 ② 근거 있는 답변 ③ 완전 로컬. 신뢰도 게이트("확인되지 않았어요") 강조 | 실제 챗 UI 스크린샷 1장 크게(답변+출처 마커) + 거절 화면 작게 |
| 5 | **아키텍처 — 어떻게 동작하나** | HTML→파싱→청크→하이브리드 인덱스→답변 5단계. 브레드크럼 보존이 검색 정확도·출처 신뢰의 핵심. 전 과정 사내 서버 완결 | 파이프라인 다이어그램(영상 아키텍처 씬 프레임 재활용) + 브레드크럼 경로 예시 칩 |

| 7 | **데모 하이라이트 + 보안·비용** | 실측 데모 시퀀스 3컷(질문→근거→답변→거절) 스트립. 보안: 데이터 무반출·오픈소스 스택·추가 라이선스 0원 | 스크린샷 3연속 스트립(영상 프레임) + 보안 배지 3종 |
| 8 | **로드맵 & 클로징** | 계좌(완료)→주문·출납→전 부문. 필요 의사결정 사항(서버 자원, 확산 일정) 1~2줄. 문의처 | 로드맵 스텝 그래픽 + 다크 클로징 밴드 |

## 4. 이미지 소스 (전부 확보됨 — 신규 촬영 불필요)

| 소스 | 용도 |
|---|---|
| `video/out/thumb.png`, 영상 씬 프레임(ffmpeg 추출) | 표지 보조, 문제정의·아키텍처·성과 씬 비주얼 |
| 웹 UI 실캡처 PNG(데모 녹화에서 프레임 추출) | 4·7쪽 스크린샷 — **실제 동작 증거** |

## 5. 기술 파이프라인 (재생산 가능, 이미 갖춘 도구만 사용)

```
docs/report/
├── content/          # 쪽별 원고 (마크다운, 섹션당 1파일)
├── template/         # HTML 템플릿 + print.css (브랜드 시스템)
├── images/           # 추출·가공된 이미지
├── build.mjs         # md → HTML 조립 → Playwright page.pdf() → A4 PDF
└── out/              # PB매뉴얼데스크_개발보고_2026-07.pdf (git-ignore)
```

- **마크다운 → HTML**: 경량 변환(기존 스타일. 의존성 최소 — marked 1개)
- **HTML → PDF**: **Playwright `page.pdf()`** — 이미 설치된 chrome-headless-shell 재사용. CSS `@page`로 A4·여백·풋터 제어, `-webkit-print-color-adjust: exact`로 다크 표지 색 보존
- **폰트**: Pretendard(레포 내 woff2) `@font-face` 임베드 — PDF에 서브셋 임베드되어 어디서 열어도 동일
- **워드/PPT 변환 대비**: 원고가 마크다운이므로 필요 시 pandoc으로 docx 파생 가능(부차 산출물)

이 방식의 장점: 영상 때 검증한 도구 체인 그대로 → **추가 설치 0**, 원고 수정 → `node build.mjs` 한 번으로 재생산.

## 6. 원고 작성 원칙 (임원 독자 기준)

- 쪽당 핵심 메시지 1개 — 섹션 제목이 곧 결론("파일럿이 아닙니다 — 계좌 전체가 준비됐습니다")
- 기술 용어는 본문에서 풀고(예: "의미 검색과 키워드 검색을 함께 쓰는") 정식 명칭(FAISS·BM25)은 캡션·부록으로
- 모든 주장에 숫자 또는 스크린샷 근거를 병기 — 영상과 동일한 "근거 있는 답변" 철학을 문서 자체가 시연
- 원고 초안은 기존 자산(README, 기획.md, 영상 자막 82개 큐)에서 재조립 — 새로 쓰는 양 최소화

## 7. 작업 단계 (총 반나절~1일)

| 단계 | 작업 | 산출물 | 공수 |
|---|---|---|---|
| P1 | 영상 프레임·UI 캡처에서 이미지 추출/선별 + 마스킹 점검 | images/ 12~15장 | 1h |
| P2 | 쪽별 원고 작성(기존 자산 재조립) | content/*.md | 2h |
| P3 | HTML 템플릿 + print.css (표지·본문·인포그래픽 컴포넌트) | template/ | 2~3h |
| P4 | build.mjs 작성 → PDF 1차 빌드 → 페이지 넘김·색상 검수 | 초안 PDF | 1h |
| P5 | 윈도우에서 실열람 검수(Acrobat/브라우저/인쇄 미리보기) → 수정 반영 | 최종 PDF | 30m |

## 8. 품질 체크리스트 (완료 기준)

- [ ] 표지 포함 8쪽 이내, 쪽당 메시지 1개
- [ ] 다크 표지 색이 PDF에서 뭉개지지 않음(print-color-adjust 확인)
- [ ] 한글 폰트 임베드 — 다른 PC에서 열어도 Pretendard 유지
- [ ] 수치 일관성: 영상·README·PDF의 356/4,443/5:5 등 숫자 일치
- [ ] 스크린샷 내 사내 IP·민감정보 마스킹
- [ ] 인쇄 테스트(흑백 출력 시에도 대비 유지 — 오렌지가 회색으로 죽지 않는 웨이트 사용)
- [ ] 파일 크기 10MB 이하(메일 첨부 기준)
