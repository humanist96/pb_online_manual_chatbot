# 업무매뉴얼 확장 계획서 — 화면/업무 이원 매뉴얼 색인과 선택

> 지금까지 색인한 것은 **화면매뉴얼**(`/000/ST/`, 19부문 4,108화면)이다. 별도의
> **업무매뉴얼**(`/000/PM/`)을 추가 색인하고, 사용자가 **화면/업무를 선택**해 질문할 수
> 있게 한다. 테스트는 로컬 우선(온라인 반영은 범위 외 — 완료 후 별도 결정).

## 1. 핵심 설계 — 매뉴얼 종류를 브레드크럼 최상위로

기존 스코프 체계를 그대로 키운다. **`sector_path`의 최상위에 매뉴얼 레벨을 추가**:

```
기존:  계좌 › 고객관리 › 고객정보조회 › 화면설명 › …
확장:  화면 › 계좌 › 고객관리 › 고객정보조회 › 화면설명 › …
       업무 › 계좌관리 › 계좌개설 › 개별계좌개설 › (문서) › (섹션) › …
```

이 선택의 효과 — **스코프 필터·캐스케이딩 셀렉터·근거 지도·"좁히기"가 무수정으로 매뉴얼 차원까지 동작**한다(전부 sector_path 접두 매칭 기반). 셀렉터 1열이 "화면/업무", 2열이 부문이 된다. "화면 매뉴얼만 볼래" = `scope=화면`, "업무의 계좌관리만" = `scope=업무>계좌관리`.

**청크 스키마** (기존 필드 하위호환):

```json
{
  "manual":      "업무",                       // "화면" | "업무" — 신규 1차 구분
  "sector":      "계좌관리",                    // 부문(기존 의미 유지 — scope_hint 부문 분포용)
  "sector_path": ["업무", "계좌관리", "계좌개설"],  // 매뉴얼 레벨이 루트로
  "embed_text":  "[업무/계좌관리] 문서제목 > … : …"   // 매뉴얼 문맥까지 임베딩에 주입
}
```

- ⚠️ 기존 화면 청크도 sector_path·embed_text가 바뀌므로 **전량 재색인 1회**(로컬 빌드 수십 분, 이미 검증된 절차). embed_text 접두는 `[화면/계좌]` 형태로 통일
- **모호성 피드백 2단계화**: scope_hint에 `manuals` 분포 추가 — "계좌 개설"처럼 화면(조작법)과 업무(업무 절차) 양쪽에 근거가 걸리면 배너가 먼저 **"화면 기준? 업무 기준?"**을 묻고, 매뉴얼 확정 후엔 기존 부문 배너가 동작

## 2. 신규 파서 — `src/parse_pm.py`

화면매뉴얼 파서(parse.py)와 템플릿 계열이 달라 **별도 모듈**로 작성(parse.py 불변 = 기존 골든 무위험).

파일럿 관찰(ACP01010) 기반 추출 규칙 초안 — **파일럿 게이트에서 확정**:
- 문서 제목 = h1. 루트 브레드크럼 = [매뉴얼, …TOC 경로…, 제목]의 문서 내 경로 부분
- `div#content` 직하 `p` = 개요(chunk_type=overview)
- `ol > li` = 섹션 단위. li 내 선두 텍스트를 섹션 라벨로, 내부 `table.box_style1`은 th 헤더행 기준 (항목, 설명) 행 분해 — 화면매뉴얼의 th/td 규칙과 유사 접근
- `s-01`/`h3` 클래스 = 소제목 후보(파일럿 표본 확대로 검증)
- chunk_type: overview / description 재사용(용어찾기·질문보기·관련화면 섹션은 PM에 없음 — 유형 필터 UI는 기존 유지, PM 청크는 overview/description으로 수렴)
- 골든 테스트: `tests/test_parse_pm.py` 신설(부문별 파일럿 2~3화면 골든 누적, 기존 test_parse.py 불변)

리스크: 한 샘플만 관찰함 — 부문별(특히 제반법규, 본사업무) 템플릿 변형 가능. **P1에서 부문당 2문서 표본(20개) 일괄 검사** 후 규칙 확정.

## 2-B. 이미지 절차 텍스트화 — 업무매뉴얼의 본체 (핵심 요구)

**실측 근거(ACP02010 "계좌개설절차")**: 본문 텍스트가 **93자(제목뿐)**이고 실제 내용은
GIF 2장 — ①계좌개설 흐름도(고객방문→신규여부 분기 TR-2100→고객정보등록→CDD등록
TR-6954→종합계좌추가 개설 TR-2105→카드발급 TR-2310→…) ②단계별 주요내용 표.
**이미지를 읽지 못하면 이 문서는 RAG에서 빈 문서다.** 흐름도는 화살표·분기 관계가
의미의 절반이므로 OCR만으로는 부족 — 구조를 이해하는 VLM이 필요하다.

**설계 — 로컬 2패스 하이브리드** (CLAUDE.md 로컬·오픈소스 제약 준수):

```
data/html_pm/*.html ─▶ 이미지 수집(crawl 확장) ─▶ data/img_pm/
                                                    │
  패스1 OCR (EasyOCR 한국어 — 기존 torch CPU 재사용)  │  노드 라벨·TR코드 고정밀 추출
  패스2 VLM (Ollama qwen2.5-vl:7b — 로컬)            ▼
    프롬프트에 OCR 결과를 주입해 환각 억제:
    "이미지의 텍스트는 다음과 같다: … 흐름도의 순서·분기 관계를
     단계별 서술로 재구성하라. 텍스트에 없는 코드·명칭을 만들지 마라."
                    │
                    ▼
  data/pm_image_text.json  (이미지 sha1 캐시 — 재실행·증분 안전)
                    │
  parse_pm.py가 이미지 위치에 병합 ─▶ 브레드크럼 [업무, …, 문서, 흐름도, 단계N]
  chunk_type=description · text에 "(도식 텍스트화: ACP02010_01.gif)" 출처 마커
```

- **처리 전략(비용 관리)**: ① OCR은 전량(이미지당 ~수 초) ② VLM은 우선순위 큐 —
  "본문 텍스트 빈약(<300자) && 이미지 보유" 문서 먼저(ACP02010류가 정확히 이 조건),
  나머지는 OCR 텍스트만으로 1차 색인 후 VLM 배치를 야간 진행(CPU 이미지당 1~3분)
- **신규 의존성**: `easyocr`(torch 재사용), Ollama `qwen2.5-vl:7b`(~6GB, 최초 1회) —
  모두 로컬 오픈소스. requirements는 본체 오염 방지 위해 `requirements-pm.txt` 분리
- **품질 게이트**: 파일럿(ACP02010 포함) VLM 출력을 사람이 검수 — TR코드 오인식률·분기
  누락 확인 후 전량 배치. 골든 테스트에 "텍스트화 청크에 TR-2105 포함" 류 assert
- (옵션, 별도 승인 필요) 개발기 한정 claude CLI 비전 배치 — 품질·속도 우위지만 사내
  이미지의 외부 전송이므로 기본안은 로컬 VLM

## 7. 성공 기준

- 사용자가 컴포저 [범위]에서 **화면/업무를 첫 단계로 선택**할 수 있다
- "계좌 개설" 같은 교차 질의에서 배너가 **"화면 기준? 업무 기준?"을 먼저 묻는다**
- `scope=업무` 검색에 화면매뉴얼 근거가 0건 (완전 분리)
- **"계좌개설 절차 알려줘"에 흐름도 기반 단계·TR코드가 답변된다** (이미지 → 텍스트화 검증의 최종 기준)
- 기존 화면매뉴얼 품질(골든 8종·실측 시나리오) 무손실
