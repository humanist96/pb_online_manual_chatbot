# PB 챗봇 UI/UX 리디자인 계획안 — "매뉴얼 데스크"

> 대상: `web/index.html` + `src/webapp.py` 프런트 전면 개편
> 목표: 사내 담당자가 매일 쓰고 싶은 **상업 서비스 수준**의 매뉴얼 Q&A 경험.
> 제약: 폐쇄망 — 외부 CDN·폰트·JS 라이브러리 금지, 모든 자산 로컬 번들. 백엔드는 표준 라이브러리 `http.server` 유지.

---

## 1. 현황 진단 — 무엇이 상업 수준을 막고 있나

지금 UI는 잘 만든 **엔지니어링 콘솔**이지 **제품**이 아니다.

| # | 문제 | 상업 서비스 관점 |
|---|------|----------------|
| 1 | α·τ·top-k 슬라이더가 첫 화면에 노출 | Perplexity는 랭커 노브를 사용자에게 절대 보여주지 않는다. 업무 담당자에게 "혼합 가중치 α"는 소음이다 |
| 2 | 좌측 폼 → 우측 결과의 "검색 폼" 패러다임 | 상업 AI 답변 제품은 전부 **하단 고정 입력창 + 중앙 답변 스레드** 구조로 수렴했다 |
| 3 | 단발 질의만 가능 — 대화·이력·후속질문 없음 | 매뉴얼 질문은 꼬리를 문다("그럼 AUP는 어디서?"). 이력이 없으면 재방문 가치가 없다 |
| 4 | 첫 화면이 "질문 대기" 스탬프뿐 | 무엇을 물어야 할지 모르는 사용자를 방치. 추천 질문이 없다 |
| 5 | 답변이 `pre-wrap` 민무늬 텍스트 | 매뉴얼 원문은 단계·표·용어 구조인데 답변은 구조를 잃는다 |
| 6 | 로딩이 점 세 개(`···`)뿐 | CPU Ollama는 수 초~수십 초. 단계별 진행 표시가 없으면 죽은 것처럼 보인다 |
| 7 | 근거 카드에 랭크 탭·배지·dense/sparse 바·신뢰도·경로·본문·푸터가 전부 동시 노출 | QA 계측이 최종 사용자 화면에 그대로 새어 나온 상태. 정보 위계 실종 |
| 8 | 840px 단일 브레이크포인트 | 모바일·태블릿에서 사실상 사용 불가 |

**유지할 자산**: KOSCOM 오렌지 아이덴티티, dense↔sparse 대비 바(QA 모드로 이동), τ 게이트("매뉴얼에서 확인되지 않습니다") 철학, [S1] 인용 칩 ↔ 근거 스크롤 연동.

## 2. 벤치마크 — 무엇을 배껴올 것인가

| 서비스 | 배껴올 것 | 우리 적용 |
|--------|-----------|-----------|
| **Perplexity** | 답변 중심 중앙 칼럼, 위첨자 인용 ↔ 출처 카드 양방향 하이라이트, 답변 위 출처 스트립, 하단 고정 후속질문 입력창, 관련 질문 제안 | 화면 구조의 뼈대 전체. 이 계획안의 기본 골격 |
| **ChatGPT / Claude** | 좌측 접이식 세션 이력, 스트리밍 체감, 답변 복사 버튼, 재생성 | 대화 스레드 + localStorage 이력 |
| **Glean** (엔터프라이즈 검색) | 출처 신뢰 마커, 필터를 체크박스가 아닌 칩으로, "출처 n건" 접기 | 청크 유형 필터 칩, 근거 패널 접기 |
| **GitBook·Mintlify "Ask AI"** | 문서 특화: 결과에 브레드크럼 상시 노출, 원문으로 점프 | 근거 카드의 경로 표기, "원문 매뉴얼 ↗" |
| **Algolia DocSearch / Linear** | ⌘K 즉시 포커스, 키보드 우선, 조용한 표면과 명확한 포커스 링 | 단축키 체계, 포커스 스타일 |
| **토스 (TDS)** | 한국 금융 서비스의 마이크로카피 톤 — 존댓말·행동 지시형·짧은 문장, 넉넉한 여백, 큰 라운드 | 전체 카피라이팅과 여백 감각. 사용자가 코스콤 직원이라 이 감성이 곧 신뢰다 |

> 원칙: **구조는 Perplexity, 톤은 토스, 계측은 숨긴 Linear.**

## 3. 컨셉 — 두 개의 모드, 하나의 제품

한 화면에 섞여 있던 두 사용자를 분리한다.

- **상담 모드 (기본)** — 업무 담당자용. 질문 → 답변 → 근거 → 후속질문. 노브 제로.
- **품질 모드 (QA)** — 엔지니어용. 헤더 토글(단축키 `Q`)로 진입. α·τ·top-k 슬라이더, dense/sparse 대비 바, 점수 테이블, 게이트 배너 등 현재 콘솔의 계측 전부가 이쪽으로 이사한다. 같은 화면 골격 위에 계측 레이어만 켜지는 방식.

**페이지의 단 하나의 일**: "이 업무, 매뉴얼 어디에 어떻게 나와 있어?"에 30초 안에 답과 *위치*를 주는 것.

## 4. 디자인 토큰

### 4.1 컬러 — "백지 위의 오렌지 원장"

KOSCOM CI(오렌지↔진회색)는 브랜드 확정값이므로 유지하되, 사용 규율을 다시 세운다.
**오렌지는 오직 세 곳**: 인용 칩, 주 행동 버튼, 근거 지도의 활성 경로. 나머지는 전부 무채색.

```
--ground     #F6F7F9   페이지 바탕 (쿨 미스트)
--surface    #FFFFFF   카드·답변·입력창
--ink        #191C20   본문
--ink-2      #4B5361   보조 텍스트
--dim        #8A93A1   메타·라벨
--line       #E6E9EE   헤어라인

--brand      #F5821F   KOSCOM 오렌지 — 칠하기 전용(버튼·칩 배경·게이지)
--brand-text #C9610A   오렌지 텍스트 전용 (흰 바탕 AA 대비 확보용 — #F5821F는 소형 텍스트 대비 미달)
--brand-tint #FFF4E9   인용 칩·하이라이트 배경
--steel      #3C434C   대비축 — 사용자 말풍선, sparse 게이지
--ok         #1E8A62   게이트 통과·신뢰 마커 (채도 낮게)
--warn       #B7791F   저신뢰 플래그
```

청크 유형 5색은 유지하되 채도를 한 단계 낮춰 배지가 아닌 **잉크 도트 + 텍스트 라벨**로 격하한다(배지 5색 나열이 현재 카드를 시끄럽게 만드는 주범).

### 4.2 타이포그래피 — 읽는 답변, 세는 코드

| 역할 | 서체 | 규격 |
|------|------|------|
| 본문·UI | **Pretendard Variable** (woff2 서브셋 로컬 번들 `web/fonts/`) | 답변 본문 16px/1.75 — "읽기" 최우선. UI 13.5px |
| 디스플레이 | Pretendard 700, letter-spacing -0.02em | 첫 화면 인사 26px, 이후 등장 없음 — 절제 |
| 데이터 | D2Coding → Cascadia Code → ui-monospace | 화면ID·점수·ms. `tabular-nums` 필수 |

**도메인 시그니처 타이포**: 화면 코드(`AC250400`)를 증권 시세판의 종목 티커처럼 취급 — 모노 + 미스트 배경 박스 + 클릭 시 복사. 매뉴얼 세계의 고유 명사를 시각적 1급 시민으로.

폐쇄망 주의: 현재 CSS는 Pretendard를 선언만 하고 로드하지 않음(시스템에 없으면 맑은고딕 폴백). **KS 완성형 2,780자 서브셋 woff2 3종(450/600/700)을 저장소에 커밋**하는 것이 P0.

- **첫 화면(빈 상태)**: 스레드 영역에 인사 한 줄("PowerBASE 계좌 매뉴얼, 무엇이든 물어보세요") + **추천 질문 칩 4개**(매뉴얼 qa 청크에서 실제 질문 추출 — 빌드 시 생성하거나 `/api/meta`에 samples 필드 추가). 컴포저는 이때만 화면 중앙에 크게 — 질문이 시작되면 하단으로 도킹 (Perplexity 패턴).
- **모바일(<840px)**: 이력·근거 패널은 시트(sheet)로. 인용 칩 탭 → 근거 시트가 하단에서 올라옴.

## 5. 시그니처 — 근거 지도 (Evidence Map)

이 제품이 기억될 단 하나의 요소. **top-k 근거들의 브레드크럼 경로를 병합한 미니 트리**를 근거 패널 최상단에 렌더한다.

이 데이터는 이미 있다 — 파이프라인의 핵심 개념이 "브레드크럼 1개 = 청크 1개"이고 모든 hit이 `section_path` 배열을 갖고 있다. 지금은 카드마다 `A › B › C` 한 줄로 낭비되고 있을 뿐이다.

```
지점계좌서비스약정등록내역               ← 공통 루트 (문서 제목)
├─ 화면설명
│   ├─ 조회 방법          [S1] [S3]    ← 인용 칩이 트리 노드에 정박
│   └─ 등록 절차          [S2]
└─ 용어찾기
    └─ 변경사용자          [S4]
```

동작:
- 답변의 위첨자 인용¹ hover → 트리에서 해당 경로가 **오렌지로 점등**되고 근거 카드가 부상. 반대 방향도 동일 (양방향).
- 트리 노드 클릭 → 해당 근거 카드로 스크롤.
- 근거가 문서 여러 곳에 흩어졌는지, 한 섹션에 몰렸는지가 **답의 신뢰도를 직관적으로** 말해준다 — 계측 없이도 품질이 보이는 장치.

원장(ledger)시스템 매뉴얼의 본질이 계층이므로, 계층 그 자체를 인터페이스로 만든다. 이것이 이 브리프에서만 나올 수 있는 선택이다.

## 6. 인터랙션 & 모션

**단 하나의 연출된 순간**: 질문 전송 → 답변 도착 시퀀스. 나머지 모션은 전부 절제.

1. 전송 → 사용자 버블이 스레드에 즉시 안착, 컴포저 비활성.
2. **단계형 상태 표시** (LLM이 느린 환경의 핵심 UX): `매뉴얼 검색 중… → 근거 4건 확보 → 답변 작성 중…` — `/api/answer`가 search_ms/gen_ms를 이미 분리 측정하므로, P3에서 검색·생성을 2회 호출로 나누면(`/api/search` 먼저 → 근거 카드 즉시 렌더 → `/api/answer`) **근거가 먼저 뜨고 답변이 따라오는** Perplexity식 체감 속도를 스트리밍 없이 구현 가능.
3. 답변 도착 → 위에서 8px 페이드 인, 인용 칩은 60ms 간격 순차 등장, 근거 지도 경로가 한 번 그려짐(SVG stroke 애니메이션 500ms).
4. hover 마이크로: 인용 칩 ↔ 카드 ↔ 트리 동시 하이라이트 120ms.

`prefers-reduced-motion` 시 전부 즉시 표시. 스켈레톤·스피너 남발 금지.

## 8. 접근성·품질 기준 (릴리스 게이트)

- 키보드: `/` 또는 `⌘K` 컴포저 포커스, `Enter` 전송, `Shift+Enter` 줄바꿈, `Q` QA 모드, `Esc` 시트 닫기. 인용 칩·트리 노드 Tab 순회 가능.
- 대비: 본문 AA. 오렌지 텍스트는 반드시 `--brand-text`(#C9610A) 사용.
- `:focus-visible` 오렌지 2px 링 전역.
- 360px까지 반응형 3단계 (>1200 / 840–1200 근거 패널 축소 / <840 시트).
- 인용 칩에 `aria-label="근거 1: 화면설명 › 조회 방법"`.

## 9. 구현 로드맵

### P0 — 자산·구조 준비 (반나절)
- Pretendard woff2 서브셋 3종 → `web/fonts/` 커밋 (폐쇄망 필수).
- `web/`을 `index.html + styles.css + app.js`로 분리.
- **`webapp.py`에 정적 파일 라우트 추가** — 현재 `/`와 `/api/*`만 서빙하므로 이것 없이는 파일 분리 불가 (`web/` 하위 화이트리스트 확장자만, 경로 탈출 방지).

### P1 — 골격 교체 (1~2일)
- 3존 레이아웃 + 하단 컴포저 + 스레드 렌더러(질문 버블/답변 카드).
- 답변 경량 마크다운 렌더(굵게·목록·표만, 자체 파서 ~60줄 — 외부 라이브러리 금지).
- 첫 화면 추천 질문 칩(1차는 정적 4개, 2차에 `/api/meta` samples).
- 기존 단발 검색 화면과 기능 동등성 확보 후 교체.

### P2 — 시그니처 (1일)
- 근거 지도: `section_path` 병합 트리 빌더 + 인라인 SVG/DOM 렌더 + 인용↔카드↔트리 양방향 하이라이트.
- 화면 코드 티커 컴포넌트(클릭 복사).

### P3 — 대화 경험 (1~2일)
- localStorage 세션 이력(좌측 레일, 검색·삭제).
- 2단 호출(`/api/search` 선렌더 → `/api/answer`)로 근거 선노출 체감 속도 개선.
- 관련 질문 제안(근거의 qa 청크에서 추출).
- (선택) `webapp.py` chunked 스트리밍 — Ollama `stream:true` 프록시.

### P4 — QA 모드 & 폴리시 (1일)
- QA 토글: 계측 레이어(α/τ/top-k, dense/sparse 바, 신뢰도, 랭크, ms) 이식.
- 반응형 시트, 접근성 감사, reduced-motion 검증, 마이크로카피 최종 교열.

**주의**: `chatbot.py`(CLI)와 `webapp.py`의 검색·프롬프트 로직이 중복 구현이므로 API를 건드릴 때 양쪽 확인 (CLAUDE.md 규약).

## 10. 자기 검증 — 이 계획이 템플릿 답안이 아닌 이유

- "AI 챗봇 UI"의 기본값(보라 그라데이션, 다크+네온, 크림+세리프)을 전부 배제 — 컬러는 브랜드 확정값(KOSCOM 오렌지↔진회색)에서 출발하고, 볼드의 사용처를 세 곳으로 계약했다.
- 시그니처(근거 지도)는 이 코드베이스의 고유 자산("브레드크럼 = 청크 = 검색 단위")에서만 도출 가능한 요소다. 다른 RAG 제품에 이식하면 의미가 사라진다.
- 스트리밍 불가·CPU 추론이라는 이 환경의 실제 약점을 "근거 선노출 + 단계형 상태"라는 구조로 상쇄했다 — 예쁜 스피너로 덮지 않았다.
- 벤치마크는 장식이 아니라 구조 차용이다: Perplexity의 골격, 토스의 언어, Linear의 키보드.
