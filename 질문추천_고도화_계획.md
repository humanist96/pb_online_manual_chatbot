# 질문 추천 고도화 계획 — 예상 질문 적중률 + 관련 질문 말풍선

2026-07-08. 대상: 온라인 데모(`deploy/online/`) 우선, 이후 사내 webapp 이식.

> **진행 상태(2026-07-08)**: 1단계 완료·배포됨. 2단계 중 Upstash 온라인 재검증 완료(1,039→881건, 84.8%) — LLM 후속질문·QA 지표 노출 잔여. 3단계 사내 webapp 이식 완료(`data/questions.json` 1,039건 로컬 직접 검증, 계측은 `data/chip_log.jsonl`).

## 1. 문제 정의

현재 추천 질문의 세 가지 약점이 사용자 만족도를 깎는다.

| 지점 | 현재 동작 | 문제 |
|---|---|---|
| 첫 화면 예상 질문 | `_static.py` META.samples — qa 청크에서 **선착순 8개** 고정 | 스코프 무관, 부문 다양성 없음, 매번 동일 |
| 답변 후 후속 질문 | 검색 hits에 qa 청크가 **우연히** 걸릴 때만 최대 2개, 폴백은 위 8개 재사용 | 노출이 불규칙하고 반복적 — "이미 본 질문" |
| 적중률 | 무검증 | 추천을 눌렀는데 게이트에 걸리면(답 못 하면) 신뢰 훼손 |

**적중률 정의**: 추천 말풍선을 눌러 실행된 질문이 게이트(τ)를 통과해 정상 답변을 받는 비율.

## 2. 설계 원칙

- **검증된 질문만 추천한다** — 질문뱅크의 모든 후보는 자기-검색(self-retrieval) 검증을 통과해야 함: 그 질문으로 검색했을 때 출처 화면이 top-3에 들고 게이트를 통과.
- **재료는 실제 매뉴얼 Q&A** — qa 청크 1,123건(화면당 1개꼴)의 질문은 매뉴얼이 스스로 답을 보증하므로 적중률이 구조적으로 높다.
- **추천은 근거에서 나온다** — 답변 후 관련 질문은 이번 답변의 근거(hit) 화면·부문에서 뽑는다(추가 API 호출·비용 0).
- **측정 없이는 개선 없음** — 말풍선发 질문에 `src=chip` 마커를 붙여 클릭 수·게이트 통과율을 Redis에 적재.

## 3. 단계별 계획

### 1단계 — 질문뱅크 + 말풍선 (즉시 구현)

1. **`deploy/online/gen_questions.py`** (로컬 실행 도구, 배포 제외)
   - `data/chunks.jsonl`의 qa 청크에서 질문 후보 추출(`section_path[-1]`이 "?"로 끝, 10~60자).
   - 업무매뉴얼(qa 없음)은 문서 제목 기반 템플릿 질문 1개/문서 보충.
   - **자기-검색 검증**: 로컬 FAISS+BM25 인덱스(`src/chatbot.py hybrid_search` 재사용)로 각 후보를 질의 → 출처 화면 top-3 포함 + dense 코사인 ≥ 로컬 τ 통과분만 채택.
   - 산출: `deploy/online/api/_questions.py` (`QUESTIONS = [{q, sid, t, sp, m}]` 정적 리스트).
   - ※ 로컬 임베더(ko-sroberta)와 Upstash 임베더(text-embedding-3-small)가 달라 **대리 검증**임 — 2단계에서 온라인 재검증.
2. **`/api/suggest` 신설** — `scope`(경로 접두 필터)·`n`·`seed` 파라미터. 부문 라운드로빈으로 다양성 보장. 벡터 호출 없음(정적 조회, 쿼터 무부담).
3. **`/api/answer`에 `related` 추가** — `_common.related_questions()`: hit 화면과 동일 화면 → 동일 부문 → 동일 매뉴얼 순으로 뱅크에서 최대 3개(현재 질문·중복 제외). 게이트에 걸린 턴에도 제공(막다른 골목 방지).
4. **프런트 말풍선 UI** (`public/app.js`·`styles.css`)
   - 첫 화면: `/api/suggest`로 스코프 연동 예상 질문 + "다른 질문 보기" 셔플.
   - 답변마다: `related`를 말풍선(꼬리 달린 버블, 순차 등장 애니메이션)으로 "이어서 물어볼 질문" 노출. 세션 내 이미 물은 질문 제외.
   - 게이트 턴: 정적 샘플 대신 `related`(근거 기반 우회 질문) 우선.
5. **계측**: 말풍선 클릭 시 `src=chip` 부착 → answer.py가 Redis에 일별 `클릭 수`/`게이트 통과 수` INCR. 적중률 = 통과/클릭.

### 2단계 — 온라인 재검증·LLM 후속질문 (실데이터 업로드 완료 후, ~7/16)

- 질문뱅크 전량을 **Upstash 실제 파이프라인으로 재검증**(하루 읽기 쿼터 내 분할) 후 `_questions.py` 갱신 — 대리 검증과의 차이 측정.
- gpt-5.4 답변 호출에 후속질문 2개를 JSON으로 함께 생성(추가 비용 ≈ 0) → 뱅크 질문과 혼합. 생성 질문도 표시 전 뱅크 매칭 또는 게이트 사전확인.
- QA 패널에 적중률·클릭율 지표 노출(Redis 카운터 조회).

### 3단계 — 사내 webapp 이식·확장

- 동일 패턴을 `src/webapp.py` + `web/`에 이식 — 사내판은 로컬 인덱스로 **직접 검증**이라 대리 검증 한계가 없음. `src/eval_questions.py`로 뱅크 적중률 회귀 평가(eval_scope.py 패턴).
- VLM 패스 완료 후 업무매뉴얼 질문 확충(도식 텍스트 기반 절차 질문).

## 4. 검증 방법

- `gen_questions.py` 실행 로그: 후보 수 → 검증 통과 수(통과율 목표 ≥ 80%, 채택분 적중률은 정의상 100%).
- 배포 후 프로덕션에서: ① `/api/suggest?scope=업무` 스코프 필터 동작 ② 임의 질문 → `related` 3개 응답 ③ 말풍선 클릭 → 정상 답변(게이트 통과) ④ Redis 카운터 증가 확인.
