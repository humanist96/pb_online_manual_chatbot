# 품질·사용성 고도화 기획 — 작업 패키지 + 임베딩 모델 분석

2026-07-08. 작성 근거: eval_scope 실측·질문뱅크 자기-검색·온라인 재검증 실패 분석(수치 본문 명기).
각 패키지(Q/U/O/E)는 Opus 에이전트에 그대로 위임 가능한 작업 지시서 형태 — 파일·함수 지시는 코드 검증 완료분.
선행 문서: 기획.md(설계 배경), 질문추천_고도화_계획.md(완료), 전부문확장_계획.md.

## Context

온라인 데모(Vercel+Upstash+gpt-5.4)와 사내 로컬(FAISS+BM25+ko-sroberta+bge-reranker, Ollama/추출형)이 동일 UX(질문 추천 말풍선·화면번호 UI·근거 지도)에 도달한 상태에서, 다음 단계의 품질·사용성 향상을 체계화하고 **임베딩 모델 업그레이드 필요성을 데이터로 판단**한다. 분석 재료: 질문뱅크 자기-검색(로컬 1,072→1,039, 96.9%), 온라인 재검증(→881, 84.8%) + 실패 유형 분석, eval_scope 교차 오염 평가, chip 적중률 계측(가동 중).

## Part A. 임베딩 모델 분석 — **결론: 지금은 교체 비권장, 판단 하네스만 상비**

### 현 구성
- 로컬 1차 검색: `jhgan/ko-sroberta-multitask`(768d, ~440MB) + BM25 하이브리드. **정밀 게이트: `BAAI/bge-reranker-v2-m3`**(m3 백본 크로스인코더, τ_rerank=0.5641).
- 온라인: Upstash 내장 임베딩(text-embedding-3-small+BM25) — **교체 대상 아님**(Upstash 관리형).
- 후보: `BAAI/bge-m3`(1024d, ~2.3GB) — rag_common에 이미 EMBED_MODEL로 교체 가능 구조.

### 실측 근거 (교체 비권장의 이유)
1. **헤드룸이 작다** — 실제 질문 분포(질문뱅크 1,072건) 자기-검색 top-3: 화면매뉴얼 99.2%, 전체 96.9%. bi-encoder를 키워 얻을 개선 여지가 한 자릿수 %p 미만.
2. **실패 원인이 임베더가 아니다** — 온라인 재검증 실패 분석: 업무 self-miss 121건 중 **113건(93%)이 같은 업무매뉴얼의 형제 문서로 유입**(예: "투신상품계좌개설 절차" → "펀드계좌개설" 문서). 제목이 거의 같은 이웃 문서 간 모호성 = 더 큰 임베더로 해결 불가 영역. 화면 miss 32건도 29건이 유사 화면 간 혼동. 게이트 분리도도 건강(통과 질문 cos p50 0.832 vs τ 0.70).
3. **정밀도는 이미 m3가 담당, Recall은 이미 만점** — 로컬 게이트가 bge-reranker-v2-m3(같은 세대 크로스인코더)라 1차 검색은 Recall만 충분하면 되는데, eval_scope 실측(2026-07-08, 현행 인덱스): **Recall@5 100%**(부문 균형 qa 76건, 스코프 유무 동일), 부문 정확도(top1) 96.1%, 동음이의 모호성 감지 90%, 화면/업무 스코프 누수 0/20·0/20. bi-encoder가 병목이라는 신호가 전혀 없음.
4. **전환 비용이 구체적으로 크다** — 44,340청크 CPU 재색인 수 시간, 질의 임베딩 지연 ~10배(CPU), 모델 2.3GB(개발기 RAM 7GB에서 리랭커와 동시 로딩 부담), `meta.json` 모델 일치 강제로 빌드·질의 동시 전환 필요.

### 작업 패키지 E (임베딩)
- **E1. A/B 판단 하네스 상비** *(난이도 M)* — ① `rag_common.py:20` INDEX_DIR 환경변수화(기본 data/index) ② `EMBED_MODEL=BAAI/bge-m3 INDEX_DIR=data/index_bgem3 python src/build_index.py`로 병렬 인덱스 ③ 신규 `src/eval_questions.py`: 질문뱅크 1,072건 자기-검색 Recall@1/3 + 게이트 분리도(정답/무관 cos 분포) + 질의 p50/p95 지연을 두 인덱스에 대해 비교표 출력. **교체 판정 기준: Recall@3 +2%p 이상 && 부문 정확도 유지 && p95 질의 ≤300ms && 서버 RAM 예산 통과 — 넷 다 만족할 때만 교체.**
- **E2. 실패 유형 직접 해소(임베더 교체보다 우선)** *(S)* — 업무 템플릿 질문 구체화(제목 외 절차 키워드 포함 재생성, VLM 텍스트 반영 후), 유사 형제 문서는 관련질문 추천에서 묶어 제시(오히려 UX 자산으로 활용).

## Part B. 품질 작업 패키지 (Q)

- **Q1. 멀티턴 문맥(질의 재작성)** *(M, 우선순위 2)* — 현재 `ask()`→`buildParams()`가 q 하나만 전송, 서버 프롬프트도 이전 턴 미참조 → "그 화면 번호는?" 같은 후속질문이 문맥 소실. 개선: 프런트가 직전 1~2턴 q(+top hit 화면명)를 `prev`로 전송, 서버가 LLM으로 독립형 질의 재작성 후 검색(LLM 부재 시 원 질의 폴백). 파일: web/app.js·src/webapp.py(contextualize_query 신설)·deploy/online/api/_common.py. 수용: "그 화면에서 수수료는?" 류 후속질문 게이트 통과.
- **Q2. 인용 무결성 검증** *(S~M, 우선순위 4)* — LLM이 존재하지 않는 [S6]을 인용하면 죽은 버튼, 무인용 답변은 표시 구분 없음. 개선: 서버가 답변 후처리로 hits 범위 밖 [Sn] 제거, 인용 0개면 경고 플래그 → 프런트 배너; `inline()`은 유효 rank만 버튼화. 파일: 양쪽 answer()·web/app.js inline()/mdlite(). 수용: 죽은 인용 버튼 0, 무인용 답변에 "근거 표기 없음" 배지.
- **Q3. 도메인 동의어/질의 확장** *(S~M)* — 사용자 표현("비번", "PB")과 매뉴얼 문어 불일치. 개선: glossary 청크(4,545건)에서 반자동 시드한 동의어 사전(data/synonyms.json) → BM25 sparse 경로에만 확장 주입(dense 불변 — 회귀 위험 최소). 파일: rag_common.py(expand_synonyms)·webapp.py search(). 수용: eval_scope Recall@5 비하락 + 표기 불일치 케이스 개선 확인.
- **Q4. 온라인 2-신호 게이트 강화** *(S)* — 온라인은 dense 코사인 단일 τ. 개선: 이미 병렬 수집하는 DENSE top1 + 하이브리드 상위 k 평균/격차 결합 복합 판정(추가 API 호출 0). 파일: deploy/online/api/_common.py search(). 수용: 질문뱅크 881건 게이트 통과 유지 + 무관질문 20종 차단율 개선.
- **Q5. LLM 후속질문 혼합(온라인)** *(S)* — gpt-5.4 답변 호출에 후속질문 2개 JSON 동시 생성 → 뱅크 관련질문과 혼합(표시 전 뱅크 매칭 또는 게이트 검증). 질문추천 계획 2단계 잔여분.
- **Q6. VLM 도식 텍스트화 완성(로컬)** *(M, 외부 의존: ollama 설치)* — qwen2.5vl:7b로 PM 이미지 483장 패스2 → 재청크·재색인 → 업무 절차 질문 품질 개선. 완료 후 E2 질문 재생성과 연계.
- **Q7. τ 자동 보정 하네스** *(S)* — 질문뱅크(정답셋)+무관질문셋으로 ROC 기반 τ 산출 스크립트화 — E1의 eval_questions.py에 통합. 수동 감 조정 탈피.

## Part C. 사용성 작업 패키지 (U)

- **U1. 답변 토큰 스트리밍(로컬 전용)** *(L, 우선순위 1)* — call_ollama가 stream:False라 CPU 7B 수십 초 동안 스켈레톤만. 개선: /api/answer를 chunked 스트림으로(http.server에서 flush로 가능, 무의존), Ollama stream:True relay + 말미에 메타 sentinel JSON, 프런트는 ReadableStream으로 누적 렌더. Vercel Python은 버퍼링이라 온라인 제외(gpt-5.4는 이미 1~2초). 수용: 로컬 첫 토큰 표시 ≤2초.
- **U2. 답변 피드백 👍/👎** *(M, 우선순위 3)* — 현재 계측은 chip 클릭뿐, "답이 도움됐나"는 미수집. 개선: turnFoot에 👍👎+사유 태그 → 로컬 data/feedback.jsonl(파일락 패턴 재사용)/온라인 Redis. 질문·게이트·backend·rank 동봉 → τ 보정·프롬프트 개선의 오답 코퍼스. 수용: 피드백 저장 + QA 패널 집계 노출.
- **U3. 컴포저 자동완성(타이프어헤드)** *(M, 우선순위 5)* — 검증된 질문뱅크가 있는데 입력 중엔 미사용. 개선: 입력 200ms 디바운스로 /api/suggest 부분일치 모드(또는 클라이언트 캐시 필터), 스코프 연동. 검증된 질문만 노출되므로 적중률 구조 보장. 수용: 2글자 입력부터 관련 질문 제안.
- **U4. 접근성 — 라이브 리전 고정** *(M)* — updateTurn이 aria-live 영역 전체 교체 → 낭독 누락/포커스 소실. 개선: 상태 텍스트를 고정 노드로 분리, 본문만 부분 교체, 완료 시 프로그램적 포커스.
- **U5. LCP/폰트 프리로드** *(S)* — Pretendard woff2 preload + 첫 화면 칩을 meta.samples로 즉시 렌더 후 suggest로 교체(현재 폴백 구조 재활용, 순서만 조정).
- **U6. QA 패널 지표 대시보드** *(S~M)* — chip 적중률(chipok/chip), 피드백 집계, 게이트 분포, 인기 질문 Top — Redis/chip_log 소스. 질문추천 계획 2단계 잔여분.
