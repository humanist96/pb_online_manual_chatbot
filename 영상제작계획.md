# 소개 영상 제작 계획서 — PowerBASE 계좌 매뉴얼 RAG 챗봇

> 지금까지 구축한 RAG 챗봇(파서 → 청크 → 하이브리드 인덱스 → 챗 UI → 배포)을
> **실습(터미널·브라우저 실캡처) + 한국어 자막**으로 설명하는 영상 제작 계획.
> 1차 목적은 **직책자 보고**(윈도우 PC에서 mp4 더블클릭으로 바로 재생), 부산출물로 실무자용 상세본을 만든다.
> 제작 도구는 **Remotion**(React 기반 프로그래매틱 비디오)을 중심으로 한다.

---

## 1. 개요

| 항목 | 내용 |
|---|---|
| 목적 | **직책자(임원·부서장) 보고**가 1차 목적 — 무엇을 만들었고, 어떤 가치가 있으며, 지금 어디까지 와 있는지를 짧고 명확하게 전달. 실무자 온보딩용 상세본은 부산출물 |
| 타깃 | ① **직책자(주 대상)** — 보고용 본편 시청 ② 실무 담당자·운영 관리자 — 상세본 |
| 산출물 | `report.mp4`(보고용 본편 3:30) + `full.mp4`(실무자 상세본 5:30) — 동일 씬 소스에서 컴포지션 2개로 렌더 |
| 재생 환경 | **윈도우 PC에서 파일 더블클릭으로 즉시 재생** — H.264 + AAC mp4, 코덱 설치·인터넷 불필요 (§2 참고) |
| 해상도 | 1920×1080 @30fps. 회의실 빔프로젝터 대비 자막·UI 텍스트 크게(본문 자막 44px 이상) |
| 음성 | **자막 중심(무나레이션) + BGM**을 기본안으로 함. 보고 자리에서 발표자가 구두 설명을 얹는 시나리오에 최적. 나레이션은 선택(§7) |
| 언어 | 한국어 자막 **영상에 번인(burn-in)** — 어떤 플레이어에서도 자막이 항상 보임. `.srt`는 참고용 부산출 |
| 배포 | **사내 한정**(보고 자료 첨부·그룹웨어). 매뉴얼 원문·사내 IP가 화면에 노출되므로 외부 공개 금지 |

## 2. 산출물 포맷 (윈도우 직재생 보장)

윈도우 10/11 기본 플레이어("영화 및 TV", Windows Media Player)가 추가 코덱 없이 재생하는 스펙으로 고정한다. Remotion 기본 출력이 이 조합이라 별도 변환 불필요.

| 항목 | 값 | 이유 |
|---|---|---|
| 컨테이너 | MP4 | 윈도우·PowerPoint 삽입 모두 호환 |
| 비디오 코덱 | **H.264 (High profile), yuv420p** | 윈도우 기본 디코더 지원. yuv420p가 아니면 일부 플레이어에서 검은 화면 — Remotion 렌더 옵션에 명시 |
| 오디오 코덱 | AAC 192kbps (BGM 없으면 무음 트랙 포함) | 무음 트랙도 넣어야 일부 플레이어 탐색 바 오작동 방지 |
| 비트레이트 | CRF 18 (≈8–10Mbps) | 터미널 텍스트·UI 글자 선명도 확보, 3:30 기준 약 200–300MB |
| 자막 | 영상에 번인 | 파일 하나만 전달하면 끝 — srt 사이드로드 불필요 |
| 검증 | 렌더 후 **윈도우 PC의 "영화 및 TV" + PowerPoint 삽입**으로 실재생 확인을 D4 체크리스트에 포함 | |

## 3. 제작 방식 결정

세 가지 후보를 비교한 뒤 **하이브리드(방식 C)** 를 채택한다.

| 방식 | 장점 | 단점 |
|---|---|---|
| A. OBS 단순 화면녹화 + 편집 | 빠름 | 재현 불가(코드 바뀌면 전체 재녹화), 오타·타이밍 실수가 그대로 남음, 자막 수작업 |
| B. 100% Remotion 애니메이션 | 완전 재현 가능 | 실제 동작 증명이 안 됨(모형이므로), 제작 공수 큼 |
| **C. 실캡처(스크립트 자동화) + Remotion 합성 ← 채택** | 캡처가 스크립트라 **코드가 바뀌면 재캡처→재렌더로 영상 자동 갱신**, 자막·다이어그램·줌 하이라이트는 Remotion이 담당 | 초기 셋업 필요 |

방식 C의 핵심: **캡처도 코드, 합성도 코드**. 영상 전체가 `make video` 한 번으로 재생산 가능해야 한다(이 저장소의 "재현 가능성" 철학과 동일).

## 4. 기술 스택 & 제작 파이프라인

```
[1] 캡처 (자동화 스크립트)                    [2] 합성 (Remotion)               [3] 산출
─────────────────────────────              ──────────────────────           ─────────
터미널: VHS(.tape) ──→ *.mp4 ─┐            React 컴포지션                    report.mp4 (보고용 3:30)
브라우저: Playwright ─→ *.webm ─┼─ assets → ├ 씬 시퀀스(<Series>)      ──→   full.mp4   (상세본 5:30)
다이어그램: Remotion 네이티브 ──┘            ├ 자막 트랙(captions.json)        captions.srt
                                           ├ 줌/하이라이트 오버레이           썸네일.png
                                           └ BGM + 페이드
```

- **터미널 캡처 — [VHS](https://github.com/charmbracelet/vhs)** (오픈소스): `.tape` 스크립트에 타이핑 속도·대기시간을 선언하면 결정적(deterministic) 터미널 영상이 나온다. `Output scene.mp4`, 폰트는 D2Coding/JetBrains Mono.
- **브라우저 캡처 — Playwright**: `recordVideo` 옵션으로 챗 UI 조작(질문 입력 → 근거 카드 → QA 모드 토글)을 스크립트로 재연·녹화. 대기 시간은 `expect(...)` 기반으로 안정화.
- **합성 — Remotion**: 씬 시퀀싱(`<Series>`), 캡처 영상 임베드(`<OffthreadVideo>`), 자막·콜아웃·아키텍처 다이어그램 애니메이션. 폰트는 저장소에 이미 있는 `web/fonts/PretendardVariable.woff2` 재사용(브랜드 일관성).
- **자막**: 씬별 `captions.json`(text, startFrame, durationFrames)을 단일 소스로 → Remotion 자막 컴포넌트 렌더 + 빌드 스크립트로 `.srt` 동시 생성.
- 정적 자산: 기존 `assets/banner.svg`, `assets/console.svg`를 인트로/아웃트로에 재사용.

> **주의**: Remotion·VHS·Playwright는 **영상 제작 도구 체인**으로 개발 PC에서만 사용한다.
> 챗봇 런타임 의존성에 추가하지 않는다(폐쇄망 제약과 무관한 별도 트리, §8 구조 참고).
> Remotion은 회사 규모에 따라 유료 라이선스가 필요할 수 있음 — 사내 사용 전 라이선스 확인 필수.
> 대안이 필요하면 완전 오픈소스인 Motion Canvas로 대체 가능(합성 레이어만 교체).

## 6. 자막 원칙

- 한 자막 = 최대 2줄 × 18자 내외, 최소 노출 1.5초. 화면 하단 안전영역(하단 96px) 고정.
- 문체는 README와 동일한 톤(간결한 해요체 금지, 명사형/평서문), 기술 용어는 원문 유지(FAISS, BM25, 브레드크럼).
- `captions.json`이 단일 소스: Remotion 렌더와 `.srt` 생성이 같은 파일을 읽는다. 자막 수정 = JSON 수정 → 재렌더.
- 터미널 씬은 자막과 별개로 **명령어 콜아웃**(현재 실행 중인 명령 상단 고정 배지)을 오버레이해 "지금 뭘 치고 있는지"를 항상 보이게 한다.
- **보고용 본편은 자막 밀도를 낮춘다**: 발표자가 구두 설명을 얹는 자리이므로 씬당 키 메시지 1~2개만. 빔프로젝터 가독성을 위해 본편 자막은 44px 이상(§1). 상세본은 단독 시청 전제라 자막을 촘촘히.

## 7. 나레이션 (선택 사항, 기본안은 미포함)

- 기본안: 무나레이션 + 저음량 BGM(저작권 무료 소스) — 사내 시청 환경(음소거 다수)에 유리하고 제작 공수 최소.
- 추가하기로 결정 시: 로컬 오픈소스 TTS(예: MeloTTS 한국어)로 자막 텍스트를 그대로 합성 → 씬 타이밍을 오디오 길이에 맞춰 재조정. 상용 TTS API는 사용하지 않는다(프로젝트 철학과 일관).

- `video/captures/out/`, `video/out/`, `node_modules/`는 `.gitignore` 추가. **렌더된 mp4는 커밋하지 않는다**(사내 데이터 포함 + 용량).

## 9. 캡처 사전 준비 (재현성 체크리스트)

1. 인덱스가 빌드된 상태(`data/index/` 존재 — 현재 완료)에서 캡처한다. 캡처 중 사내망 접근 불필요.
2. 데모 질문 3~4개를 사전 검증해 고정: 정답이 잘 나오는 질문 2개(예: AC250400 서비스약정 관련) + 근거 매칭이 시각적으로 보기 좋은 질문 1개 + 매뉴얼 밖 질문 1개(거절 시연).
3. LLM 백엔드 통일: 데모 일관성을 위해 `LLM_BACKEND=ollama` 고정(응답 시간 예측 가능). Ollama 응답 대기 구간은 Remotion에서 타임랩스 처리.
4. 터미널 환경 고정: 창 크기 120×30, 프롬프트 간소화(PS1), `HF_HUB_OFFLINE=1`로 다운로드 로그 방지.
5. **민감정보 마스킹**: 사내 서버 IP(`211.255.203.234`)가 crawl 로그·URL에 노출되지 않도록 해당 씬에서 제외하거나 Remotion 블러 오버레이 처리. 매뉴얼 본문 노출은 사내 배포 한정이므로 허용.
