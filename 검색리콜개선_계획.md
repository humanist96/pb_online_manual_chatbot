## 0. 배경 — 무엇이 문제인가

A/B 블라인드 평가에서 우리(A)의 약점은 **답변 생성이 아니라 검색**으로 확정됐다.

| 지표 | A (매뉴얼 데스크) | B (PB AX PoC) |
|---|---|---|
| 근거적중@3 — T1 검증 질문 | 87% | **97%** |
| 근거적중@3 — T2 심층 질문 | 60% | **80%** |
| 근거 충실성(환각 억제) | **우세** (T2 85%) | 열세 (T2 65%) |
| 응답 속도 p50 | **2.3s** | 6.3s |

패배 문항의 전형(Q32·Q33): *"선물옵션VIP수수료율지정 화면에서 단계3을 설명해줘"* → 매뉴얼에 답이 분명히 있는데 검색이 유사 화면(수수료등급지정 등)의 청크를 가져왔고, LLM은 (환각하지 않도록 훈련된 대로) **정직하게 거부**했다. 즉 **정답 문서를 상위에 올리지 못하면, 좋은 생성기가 오히려 "확인되지 않습니다"를 내놓는다** — 리콜이 전체 품질의 상한이다.

## ① 질의 재작성(Query Rewrite) 도입

### 개념

사용자는 구어체로 묻고("비번 틀려서 잠겼는데 어떻게 풀어요?"), 매뉴얼은 문어체로 쓰여 있다("비밀번호 오류 횟수 초기화"). 검색엔진은 **글자·의미가 겹쳐야** 문서를 찾는데, 이 어휘 간극(lexical gap)이 크면 정답 문서가 있어도 못 찾는다. 질의 재작성은 검색 직전에 작은 LLM이 질문을 **매뉴얼이 쓰는 표현으로 번역**해 주는 전처리다 — 오탈자 교정, 조사 제거, 축약어 확장("비번"→"비밀번호"), 핵심 키워드 추출까지. B는 이미 이 단계를 갖고 있다(응답의 `rewritten_query`·`confidence` 실측).

### 설계

- **위치**: `deploy/online/api/_common.py`의 `search()` 앞단 1콜. `/api/search`·`/api/answer` 공통 적용.
- **모델 분리**: env `OPENAI_REWRITE_MODEL`(기본 `gpt-4o-mini`) — 답변용 gpt-5.4와 분리해 비용·지연 최소화. 프롬프트: "증권 원장시스템 매뉴얼 검색용으로 질문을 정규화하라. JSON `{rewritten, keywords[]}`만 출력. 의미 추가·추측 금지."
- **적용 방식**: 하이브리드 본검색과 **게이트 dense 쿼리 모두 rewritten 기준**(원질의로 게이트를 재면 리콜 개선분이 게이트에서 잘림). 원질의는 로그·표시용 보존.
- **안전장치(강등 무해 원칙)**:
  - 타임아웃 2초 / 오류 / JSON 파싱 실패 → **원질의로 폴백**(기능 없던 때와 동일).
  - Redis 캐시 `rw:{sha1(정규화(q))}` TTL 7일 — 추천 칩 클릭·반복 질문은 재작성 0ms.
  - 요청 파라미터 `rewrite=0`, env `REWRITE_ENABLE=0`으로 즉시 차단 가능.
  - rewrite 콜은 답변 일일 상한(`ANSWER_DAILY_LIMIT`)과 **별도**(검색만 하는 1단 호출에도 발생하므로 상한을 오염시키지 않게 독립 카운터로 계측만).
- **투명성**: 응답에 `rewrite: {original, rewritten, cached}` 노출 → QA 패널에 "재작성: …" 1줄 표시(계측 콘솔 역할 유지).
- **지연 예산**: +0.3~0.8초 예상. 목표 p50 ≤3.1s(현 2.3s) — 여전히 B(6.3s)의 절반 이하.

### 변경 파일

`api/_common.py`(rewrite 함수 + search 연결 + Redis 캐시), `api/search.py`·`api/answer.py`(파라미터/응답 필드), `public/app.js`(QA 패널 표시 1줄).

### 리스크와 방어

| 리스크 | 방어 |
|---|---|
| 지연 증가 | 경량 모델 + 캐시 + 2s 타임아웃 폴백 |
| 재작성이 의미를 바꿈(질문 왜곡) | "의미 추가 금지" 프롬프트 + QA 패널 노출로 관찰 가능 + rewrite=0 우회 |
| T3 함정 질문을 그럴듯하게 "복원"해 게이트 우회 | OOS는 재작성해도 매뉴얼과 무관 — τ 게이트가 최종 방어. **T3 거부 10/10 유지가 승격 조건** |
| 비용 | gpt-4o-mini ≈ $0.0001/질의 + 캐시 적중분 0 |

## ② 검색 파라미터·재랭킹 재검토 (융합 알고리즘 + 정확매치 부스트)

### 개념

하이브리드 검색은 **의미 검색(dense)** 과 **키워드 검색(sparse/BM25)** 의 두 순위표를 하나로 합친다. 합치는 방법이 융합 알고리즘이다:
- **DBSF**(Distribution-Based Score Fusion): 두 점수의 분포를 정규화해 **점수 크기**로 합산 — 점수가 신뢰할 만할 때 유리.
- **RRF**(Reciprocal Rank Fusion): 점수를 버리고 **순위 역수**(1/(k+rank))로 합산 — 점수 분포가 들쭉날쭉해도 강건.

현재 DBSF는 초기 구축 때 비교 실측 없이 채택된 값이다. 또 하나: 지금은 융합 결과를 그대로 쓰는데, 질문에 **화면번호(4자리)·TR번호·완전한 화면명**이 명시돼 있으면 그 문서가 정답일 확률이 압도적이다 — 이런 "정확 신호"를 융합 점수보다 우선시키는 **결정적 부스트**(규칙 기반 재랭킹)가 Q32류 패배의 직접 해법이다.

### 설계

**(1) 오프라인 파라미터 스윕 — 무배포·저비용, 가장 먼저 실행**

`deploy/online/ab_eval/sweep_search.py` 신설(로컬 도구):
- 평가셋: ab_eval T1+T2 40문항(gold 문서 확정) + 질문뱅크 881에서 층화 200문항.
- 그리드: `{DBSF, RRF}` × `{부스트 on/off}` × (2단계 후) `{rewrite on/off}` → **hit@3 표** 산출.
- Upstash 쿼리만 사용(240문항 × 4조합 ≈ 1K 쿼리 — 일 쿼터 내, 필요시 분할).
- 산출 기준으로 기본값 확정 → env `FUSION_ALG`로 반영.

**(2) 정확매치 부스트 — 서버 후처리 재랭킹 (LLM 불요, 결정적)**

`_common.py`에 `boost_exact(q, hits)`:
- 질문에서 신호 추출: `TR-?(\d{4})` / 단독 4자리 숫자 / 인덱스에 존재하는 화면명 완전 포함(제목 사전은 `_static.py` SECTORS에서 로드 가능).
- 신호와 `screen_no` 또는 `title` 이 **완전 일치**하는 청크를 융합 순위 위로 승격(안정 정렬 — 일치군 내부는 기존 순서 유지).
- 보수 원칙: 부분 일치·유사 일치는 부스트하지 않음(오탐 방지).

**(3) topK 정책**

- Upstash 요청 `topK: 10`으로 상향(재랭킹 후보 풀 확보). **LLM 컨텍스트·UI 기본은 상위 5 유지**(비용·화면 불변). 근거지도는 그대로 topk 파라미터 존중.

### 변경 파일

`api/_common.py`(search 파라미터화 + boost_exact), `ab_eval/sweep_search.py`(신설), `.env.local`·Vercel env(`FUSION_ALG`).

### 리스크와 방어

| 리스크 | 방어 |
|---|---|
| 부스트 오탐(무관 숫자 우연 일치) | 완전 일치만·4자리 화면번호는 "화면/TR 문맥어 동반 시"로 한정 가능(스윕에서 검증) |
| RRF 전환이 기존 강점 훼손 | 스윕은 전 트랙 측정 — T1 하락 시 채택 안 함(데이터로 결정) |
| 게이트 τ와의 상호작용 | 게이트는 별도 DENSE 쿼리 코사인이라 융합 변경과 독립 — τ 0.70 유지 |

## ③ 심층 절차용 embed_text 보강 + 목차 청크

### 개념

dense 검색의 상한은 **임베딩에 어떤 텍스트를 넣었는가**가 정한다. 실측으로 확인된 공백 두 가지:

(a) **화면번호 부재** — 현재 embed_text는 `[화면/선물결제] 선물옵션VIP수수료율지정 > 화면설명/사용방법 > 단계1 : …` 형태로 제목·경로·본문은 있지만 **화면번호(1613)가 없다**. 실무자는 "1613 화면", "5634 수수료율"처럼 번호로 묻는 일이 흔한데, 이 질의는 dense·sparse 모두에서 불리하다.

(b) **문서 단위 착지점 부재** — "X화면 사용법을 단계별로 알려줘" 같은 질의는 특정 단계 청크가 아니라 **문서 전체**를 겨냥한다. 지금은 이 질의가 여러 step 청크와 부분 매치로 분산돼 유사 화면들에 밀린다. 화면당 1개의 **목차 청크**(제목 + 화면번호 + 별칭 + 섹션/단계 목차)를 만들면 문서 단위 질의의 확실한 착지점이 생기고, ②의 정확매치 부스트와 결합해 "화면명+단계N" 질의를 해당 화면으로 끌어온다.

### 설계 (온라인 전용·증분 — 44K 전량 재업서트 회피)

**(1) 목차 청크 생성기** — `deploy/online/gen_toc_chunks.py` 신설(로컬 도구, 배포 제외):
- 입력 `data/chunks.jsonl` → 화면/업무 문서별로 section_path 목차 수집.
- 청크 형태: id `toc:{screen_id}`, `chunk_type:"overview"`, embed_text =
  `"[화면/선물결제] 선물옵션VIP수수료율지정 (화면번호 1613 · TR1613 · TR-1613) 목차: 화면개요 / 화면설명·사용방법: 단계1 입력창, 단계2 …, 단계3 … / 용어찾기 / 관련화면"`
- text = 같은 내용의 사람이 읽는 형태(근거 카드에 표시될 때 자연스럽게).
- 산출 `data/chunks_toc.jsonl` (예상 ~1,600건 = 화면 문서 수 + 업무 문서 수).

**(2) 업서트 연결** — `ingest_real.py`의 연결 로드에 `chunks_toc.jsonl` 1줄 추가(상담매뉴얼과 동일한 **증분 오프셋 방식** — 기존 53,463 뒤에 붙어 신규분만 업로드).

**(3) 기존 청크 embed_text 전면 개정(화면번호 포함)은 후속 결정** — 결정 기준: 1~3단계 적용 후 ab_eval 재측정에서 **T2 hit@3가 80% 미만이면** 화면/업무 description 청크의 embed_text에 화면번호를 넣어 전량 재업서트(53K, 1~2일 분할). 목표 도달 시 생략(비용 회피).

### 변경 파일

`gen_toc_chunks.py`(신설), `ingest_real.py`(1줄), `.vercelignore`(생성기 제외).

### 리스크와 방어

| 목차 청크가 일반 질의에 과다 노출(본문 대신 목차가 근거로) | 목차는 본문 세부와 어휘가 달라 세부 질의엔 자연 열세 + 답변 프롬프트가 근거 내용만 사용 — ab_eval T1 회귀로 확인 |
| 상담매뉴얼과 불균형 | 상담은 Q&A 단위라 목차 불필요(문서=쌍) — 대상 아님을 명시 |

## 실행 순서·승격 기준·측정

```
1단계  ② 스윕(무배포) ──→ 융합·부스트 확정 ──→ 배포 ──→ ab_eval A-only 측정
2단계  ① rewrite 구현 ──→ 플래그 on 배포   ──→ ab_eval A-only 측정 (+스윕 rewrite축 재실행)
3단계  ③ 목차 청크 업서트                  ──→ ab_eval A-only 측정
후속   T2<80%면 embed_text 전면 개정 결정
```

- **측정 도구**: `ab_eval/run_ab.py`에 `--only-a` 모드 추가 — B 재호출 없이 동결 질문셋 60으로 A 전후 비교(B 세션 불요). 채점은 기존 `score_auto.py` 재사용.
- **승격 기준(각 단계 공통)**: 개선 지표 상승 **and** T3 거부 10/10 유지 **and** p50 ≤3.5s. 하나라도 깨지면 롤백(env 플래그라 즉시).
- **최종 목표**: T1 hit@3 87→**95%+**, T2 60→**80%+**, T4·충실성·속도 우위 유지.

## 비용 요약

| 항목 | 예상 |
|---|---|
| 스윕 | Upstash 쿼리 ~1K회(무료 쿼터 내), LLM 0 |
| rewrite 운영 | 질의당 gpt-4o-mini ~$0.0001 + 캐시 적중분 0, 지연 +0.3~0.8s |
| 목차 청크 | 업서트 ~1,600건 1회(쿼터 내), 저장 증가 ~3% |
| 구현 | ② 반나절, ① 반나절, ③ 반나절 (각 단계 측정 포함 1일 내) |
